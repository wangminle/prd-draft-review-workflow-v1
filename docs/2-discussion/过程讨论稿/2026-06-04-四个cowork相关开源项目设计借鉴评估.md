# 四个 cowork 相关开源项目设计借鉴评估

> 日期：2026-06-04
> 范围：评估 `AGI-is-going-to-arrive/workflow-cookbook`、`OpenCoworkAI/open-cowork`、`pewdiepie-archdaemon/odysseus`、`onyx-dot-app/onyx` 对“需求业务工作流 cowork 版工具”的借鉴价值。
> 结论：四个项目都值得参考，但应分层吸收。`workflow-cookbook` 参考工作流方法论，`open-cowork` 参考本地 Agent 产品化与人在环控制，`odysseus` 参考 Web 化自托管 AI 工作台与高权限能力治理，`onyx` 参考企业知识库、连接器和权限检索。

---

## 1. 总体结论

这四个项目对应的是 cowork 工具远期架构中的四个不同层面：

| 项目 | 更像什么 | 对本项目的主要价值 | 不建议照搬的部分 |
| --- | --- | --- | --- |
| `workflow-cookbook` | 确定性 Agent 工作流方法论 | Workflow/Phase/Step、结构化输出、预算、续传、并行/流水线、对抗验证 | 不要把本系统改成 Claude Code Workflow 式 CLI 或通用流程教程 |
| `open-cowork` | 本地优先 AI Agent 桌面产品 | 工作区授权、沙盒执行、Skills/MCP、Trace Panel、远程控制、审批面板、多模型配置 | 不要直接迁移到 Electron 桌面形态，也不要 MVP 阶段上 VM 级沙盒 |
| `odysseus` | Web 化自托管 AI workspace | 一体化工作台信息架构、Deep Research 报告流、文档协作、移动端/PWA、自托管默认安全设置、管理员高权限工具治理 | 不要把当前产品直接扩成邮件/日历/个人效率全家桶，也不要默认向普通成员开放 shell/file/webhook 类高权限工具 |
| `onyx` | 企业知识搜索与 Agent 平台 | 连接器、权限感知检索、Document Set、Agent/Action/MCP、查询审计、使用分析 | 不要近期直接搬 PostgreSQL/Redis/MinIO/Vespa/OpenSearch/worker 的重型栈 |

映射到本项目，最合理的吸收顺序是：

1. **P0/P1**：先从 Onyx 学团队资料库数据模型和权限过滤；从 open-cowork 学工作区授权、工具调用日志和人工确认；从 `odysseus` 学 Web 工作台信息架构、默认安全设置和管理员高权限治理；从 workflow-cookbook 学 ReviewRun 的 phase/step 结构化状态。
2. **P2**：补轻量连接器、混合检索 POC、AgentRun/ToolCallTrace、Deep Research 式外部资料收集与可视化报告。
3. **P3 以后**：再评估服务化向量库、远程审批通道、MCP 管理台、复杂沙盒、连接器权限同步、质量评估与对抗验证。

一句话：**它们不是替代当前系统的底座，而是分别补强“流程确定性、Agent 产品化、工作台集成与权限治理、知识权限层”四个缺口。**

## 2. 与当前规划的对应关系

当前 cowork 版工具规划已经确立四个核心抽象：

1. `Space`：个人空间、团队空间、项目空间。
2. `Knowledge`：团队资料、需求文档、评审规则、历史结论、个人知识。
3. `Agent`：个人 Agent、团队 Agent、审查 Agent。
4. `Workflow + Collaboration`：发起审查、AI 初审、人工确认、讲解产物、评论通知、评审会归档。

四个项目可分别补强这些抽象：

| 本项目抽象 | `workflow-cookbook` | `open-cowork` | `odysseus` | `onyx` |
| --- | --- | --- | --- | --- |
| Space | 用显式状态和 checkpoint 支持可续传流程 | 工作区目录授权、本地沙盒、工作区模板 | 单一 Web workspace、模块化页面、移动端/PWA、local-first 部署默认值 | 用户组、文档集、连接器访问范围 |
| Knowledge | 每个 phase 产出结构化知识制品 | 记忆、Skills 产物、会话上下文 | Deep Research 报告、文档协作、记忆/技能向量+关键词检索 | 企业连接器、Document Set、权限感知检索、索引状态 |
| Agent | 多 Agent 并行、pipeline、对抗验证 | 个人桌面 Agent、Trace、工具执行、远程控制 | Web Agent + research + document editing + reminder/task 联动 | Agent + knowledge + Actions，按任务配置知识源和工具 |
| Workflow | Workflow/Phase/Step、预算、恢复、Schema | 任务会话、审批、人控制、远程触发 | 多步研究任务、来源归集、可视化产物输出 | 查询、检索、Agent Action、审计分析 |
| Collaboration | 人作为 gate 和 validator | 飞书/Slack 远程控制、审批面板 | 文档由人主写、AI 辅助编辑；管理员高风险能力治理 | 共享 Agent、用户组权限、反馈与分析 |

## 3. `workflow-cookbook`：确定性工作流方法论

### 3.1 项目定位

`workflow-cookbook` 更像一本 AI 工作流 cookbook。它不是完整业务系统，也不是我们可以直接嵌入的运行时。它的价值在于把 Agent 任务从“一次聊天”拆成可复用、可恢复、可验证的流程单元。

对本项目来说，它对应的是工作流层，而不是产品界面或知识库层。

### 3.2 可借鉴点

| 可借鉴点 | 具体含义 | 映射到本项目 |
| --- | --- | --- |
| Workflow 作为可复用流程配方 | 把复杂任务拆为固定入口、阶段、步骤、产物和验收条件 | `WorkflowTemplate`、审查模板、讲解生成模板、评审会包模板 |
| Phase/Step 分层 | 长任务不要只有 running/completed，而要有明确阶段 | `ReviewStageExecution`、`AgentRun`、`AgentStep` |
| Schema 输出 | 每个阶段输出结构化结果，不只输出 Markdown | 风险项、证据、建议、置信度、阻断状态、责任角色 |
| Parallel / Pipeline | 独立角色并行执行，依赖阶段串行汇总 | 产品/技术/测试/数据/合规多角色并行评审，再由汇总 Agent 归并 |
| 预算控制 | 限制最大步骤数、token、模型成本、运行时长 | `AgentRunBudget`、模型调用成本日志、超限中止策略 |
| Checkpoint / Resume | 长流程中断后能从阶段状态继续 | `WorkflowRun.status`、`AgentStep.output_ref`、局部重跑 |
| Adversarial Validation | 用第二视角审查第一轮结果 | 复核 Agent、规则冲突检查、评审结论二次校验 |
| Workflow 库版本化 | 流程模板也需要版本、变更记录和回滚 | `WorkflowTemplate.version`、审查规则版本引用 |

### 3.3 建议新增或预留的领域对象

| 对象 | 作用 | MVP 处理方式 |
| --- | --- | --- |
| `WorkflowTemplate` | 内置流程模板，例如需求初审、评审会包生成、讲解稿生成 | 先做内置枚举，不开放用户自定义 DSL |
| `WorkflowRun` | 某次流程运行实例 | 可先并入 `ReviewTask` / `ReviewRequest`，字段预留 |
| `WorkflowPhase` | 流程阶段定义 | 先用配置常量维护 |
| `WorkflowStepRun` | 单步执行记录 | 可复用 `AgentStep` 或新增 `ReviewStageExecution` |
| `WorkflowArtifact` | 阶段产物引用 | 指向 `runtime/results` 或 `runtime/artifacts` |
| `WorkflowBudget` | 运行预算和停止条件 | P1/P2 再做成本统计与超限提醒 |

### 3.4 不建议照搬

1. 不要把本产品做成纯 CLI 或 Markdown Workflow 执行器。
2. 不要 MVP 阶段开放复杂 DSL、模板市场或用户自定义工作流。
3. 不要让 Workflow 替代业务对象。需求、审查、资料库、权限、评论、审批仍然是主模型。
4. 不要过度依赖 LLM 自述状态。阶段推进应由系统状态、产物存在性、Schema 校验和人工确认共同决定。

### 3.5 对本项目的落地建议

近期最应该吸收三件事：

1. `ReviewRun` 从单一状态升级为阶段化状态，例如 `prepare_context -> retrieve_knowledge -> role_review -> aggregate -> human_confirm -> publish_artifact`。
2. 每个审查角色输出固定 Schema，例如 `risk_items[]`、`evidence_refs[]`、`blocking_decision`、`confidence`。
3. 每次 Agent/Skill 执行记录预算、输入、输出、错误、可重试标记，方便恢复和审计。

## 4. `open-cowork`：本地 Agent 产品化与人在环控制

### 4.1 项目定位

`open-cowork` 是一个开源 AI Agent 桌面应用，面向 Windows 和 macOS，强调一键安装、多模型、VM 级沙盒、Skills、MCP、GUI 自动化和飞书/Slack 远程控制。

它对本项目的价值不是桌面形态，而是“如何把一个能真实操作文件和工具的 Agent 做成用户可控、可观察、可审批的产品”。

### 4.2 可借鉴点

| 可借鉴点 | 具体含义 | 映射到本项目 |
| --- | --- | --- |
| 工作区授权 | 用户先选择/授权 AI 可操作的工作区范围 | 个人空间、团队空间、项目空间的资源边界 |
| 沙盒执行 | 文件、命令、工具调用不直接暴露宿主环境 | P0/P1 做路径白名单和操作审计；中后期评估容器/VM |
| Skills 系统 | 把文档、表格、PPT、PDF 等能力封装成可管理技能 | 现有 SkillRunner 升级为 `SkillPackage` 管理 |
| MCP 管理 | 通过 MCP 连接浏览器、Notion、桌面应用等外部工具 | 后续连接器/MCP 管理台、工具白名单、权限授权 |
| Trace Panel | 实时查看 AI 推理、工具调用、执行结果 | `AgentRunTrace`、`ToolCallTrace`、审查执行轨迹 |
| 人在环审批 | 文件修改、删除、外部发送等高风险动作需确认 | `AgentApprovalRequest`、关键动作二次确认 |
| 飞书/Slack 远程控制 | 用户可从 IM 通道发起任务、接收结果、批准动作 | 后续飞书/企业 IM 通知与审批通道 |
| 多模型配置 | 支持多家模型和 OpenAI 兼容接口 | 复用现有 `ModelConfig`、thinking adapter、模型选择器 |
| 工作区模板/记忆 | 复用成功任务上下文和配置 | 团队 Agent 模板、审查模板、项目知识快照 |

### 4.3 建议新增或预留的领域对象

| 对象 | 作用 | MVP 处理方式 |
| --- | --- | --- |
| `WorkspacePermission` | 空间/目录/资料的读写执行授权 | 先按空间资源做粗粒度权限，不开放任意目录 |
| `ToolCallTrace` | 每次工具调用的输入、输出、耗时、错误 | P1 就应该记录，便于排查和审计 |
| `AgentApprovalRequest` | Agent 高风险动作的人审请求 | P1/P2 对“通知、归档、修改资料”做确认 |
| `SkillPackage` | 技能包元数据、版本、适用场景、权限声明 | 先把现有 skills 做元数据登记 |
| `ConnectorConfig` | MCP/外部工具配置 | 后续与知识源连接器统一治理 |
| `SandboxPolicy` | 工具可访问路径、网络、命令范围 | MVP 先做白名单和禁止项，容器/VM 后置 |

### 4.4 不建议照搬

1. 不建议把当前 FastAPI + Web 工具迁移成 Electron 桌面应用。
2. 不建议 MVP 阶段实现 WSL2/Lima 级 VM 沙盒，会明显拉高部署和维护成本。
3. 不建议近期做 GUI 自动化。需求评审业务的核心不是操控桌面，而是知识、审查、协作和审批。
4. 不建议把远程 IM 控制作为第一入口。早期可以先做通知和确认，任务发起仍以 Web 界面为主。

### 4.5 对本项目的落地建议

近期最应该吸收四件事：

1. 所有 Agent 写操作都要显式落日志，不能只有最终回答。
2. 所有高风险动作都要进入 `approval_required` 状态，由具体人确认。
3. 工具能力要有“可见、可开关、可审计”的管理面，不要只藏在 Prompt 里。
4. 个人 Agent 不能默认代表本人对外承诺，应该优先生成待确认回复。

## 5. `odysseus`：Web 化自托管 AI 工作台与高权限能力治理

### 5.1 项目定位

`odysseus` 把自己定义成“自托管版 ChatGPT / Claude UI 体验”，但不是单纯聊天壳，而是一个集成 `Chat`、`Agent`、`Deep Research`、`Documents`、`Memory / Skills`、`Notes & Tasks`、`Calendar`、`Email` 的 AI workspace。

对本项目来说，它最有参考价值的不是“功能多”，而是两点：

1. **它是 Web 化、自托管、默认本地优先的 AI 工作台**，比 `open-cowork` 的桌面/VM 形态更接近我们当前 FastAPI + Web 的产品路径。
2. **它明确把自己当成高权限管理台来对待**，对 `AUTH_ENABLED`、`127.0.0.1` 默认绑定、管理员高权限工具、`data/` 与密钥不入 git 做了很明确的安全约束，这和我们当前的 `runtime/` 数据隔离原则高度同向。

### 5.2 可借鉴点

| 可借鉴点 | 具体含义 | 映射到本项目 |
| --- | --- | --- |
| 一体化 AI 工作台 | 把聊天、Agent、研究、文档、任务放在一个统一 workspace 中，而不是多个孤立工具 | “团队空间”后续可以承接资料库、审查、外部调研、讲解产物，而不是只做一个上传页 |
| Deep Research 产物化 | 多步 gather/read/synthesize 后输出可视化报告，而不是只给长回答 | 可演进为“外部竞品/规范调研 -> 证据包 -> 审查补充材料/讲解稿” |
| 文档协作理念 | README 明确强调“YOU write the text, AI assists, not the opposite” | 适合我们的审查、PRD 草稿、讲解稿场景：人负责最终责任，AI 负责补强与建议 |
| 记忆/技能检索 | `ChromaDB + fastembed + vector + keyword retrieval` 的轻量记忆层 | 可参考“个人 Agent / 团队 Agent 的记忆层”设计，但近期不必直接照搬 ChromaDB |
| 任务/提醒联动 | Notes、Todo、scheduled tasks 可触发 agent 行为，并可走 browser/email channel | 后续消息中心、待确认回复、提醒/催办机制可吸收其任务驱动思路 |
| 移动端/PWA | 同一套 Web 界面兼容手机与桌面，并支持安装 | 对我们“团队空间 / 审查工作台 / 管理后台”统一 Web 体验有参考价值 |
| 默认安全部署 | 默认绑 `127.0.0.1`、建议 HTTPS + 反代、强调不要直接暴露公网 | 可作为我们内网单机部署、反向代理、安全默认值的参考范式 |
| 管理员高权限治理 | 非 admin 默认不给 shell/Python/file read-write；MCP 管理、API token、webhook、settings 等为 admin-gated | 很适合我们后续 Agent/MCP/连接器能力的分级授权与高风险动作治理 |
| 运行时数据分离 | `data/` 下存数据库、上传、记忆、向量索引，且默认 gitignored | 和我们 `runtime/` 原则一致，可强化“索引、缓存、上传、产物、令牌一律运行时落盘”约束 |

### 5.3 建议新增或预留的领域对象

| 对象 | 作用 | MVP 处理方式 |
| --- | --- | --- |
| `WorkspaceSurfaceConfig` | 管理团队空间内各一级/二级模块是否开启、显示顺序和入口文案 | 先做前端常量或系统配置，不开放用户自定义拖拽 |
| `ResearchRun` | 外部资料调研、多步收集、归纳、生成报告的运行实例 | P2/P3 再做，不进 P0 核心范围 |
| `ResearchArtifact` | 调研报告、来源列表、截图/引用证据包 | 指向 `runtime/results`，与审查报告同级管理 |
| `UserToolPrivilege` | 用户级工具能力开关，如 web_search / mcp / file_write / external_notify | 与角色权限并存，先少量内置布尔权限 |
| `NotificationEndpoint` | 浏览器/邮件/IM 等通知出口配置 | 先预留，近期只做站内提醒或手工确认 |
| `PersonalMemoryProfile` | 个人 Agent 的偏好、记忆、技能快照 | P3 以后考虑，MVP 不做自动长期记忆扩张 |

### 5.4 不建议照搬

1. 不要把当前产品直接扩成邮件、日历、笔记、任务、文档编辑的全家桶。我们的主线仍然是需求审查协作。
2. 不要因为它用了 `ChromaDB` 就直接替换我们当前轻量栈。近期仍应以 SQLite FTS5 + 小规模向量 POC 为主。
3. 不要向普通成员默认开放 shell、文件写入、webhook、外部邮件发送这类高权限能力。
4. 不要让“Memory”变成无边界的长期业务信息黑盒。企业资料、个人偏好、审查记录、外部证据仍应分层管理。
5. 不要把“Web 工作台功能多”误解为“路线更适合 MVP”。它的广谱能力值得参考，但我们只能吸收其中对审查协作有直接价值的那部分。

### 5.5 对本项目的落地建议

近期最应该吸收五件事：

1. “团队空间”不只是一页资料库，而应逐步演进为资料、调研、审查、讲解产物的统一工作台。
2. 高风险工具能力必须做 **admin-gated + per-user privilege** 双层控制，而不是仅靠前端隐藏按钮。
3. 后续外部调研能力应优先做“ResearchRun + 引用来源 + 报告产物”，而不是只返回一段聊天文本。
4. 部署默认值应继续坚持“本地绑定、认证开启、反代暴露、运行时数据不入 git”。
5. 人写文档、AI 辅助编辑的责任边界要写进产品心智，尤其在 PRD 草稿、审查结论、讲解稿场景中不能反过来。

## 6. `onyx`：企业知识库、连接器与权限感知 RAG

### 6.1 项目定位

`onyx` 是成熟的企业知识搜索和 Agent 平台。它的主线是把组织内部数据源通过连接器接入索引，再让用户和 Agent 在权限边界内检索、问答和执行 Actions。

它对本项目的价值最高的部分，是“团队资料库和权限检索”。

### 6.2 可借鉴点

| 可借鉴点 | 具体含义 | 映射到本项目 |
| --- | --- | --- |
| Connector + Credential | 知识源连接器和凭证分离 | 飞书、Notion、GitHub、文件夹、API 导入 |
| Sync Status | 每个连接器有同步状态、刷新频率、错误记录 | `SyncJob`、`IndexAttempt`、`KnowledgeSource.status` |
| Document Access Control | 数据源可设 Private/Public/Auto Sync Permissions | 团队资料库权限、项目资料权限、个人知识私有 |
| Permission Sync | 从外部系统同步访问权限 | 中后期飞书/GitHub/Notion 权限同步 |
| Document Set | 把多个连接器/资料集合成可检索范围 | 团队规则集、项目资料集、审查规则集 |
| Agent scoped knowledge | Agent 可绑定特定知识集合 | 团队 Agent、审查 Agent、项目 Agent |
| Actions + MCP | Agent 可调用内部搜索、Web Search、代码执行、MCP/OpenAPI 工具 | ToolRegistry、MCPAdapter、Action 白名单 |
| 混合检索 | 不只做向量检索，也结合全文、过滤、元数据和重排 | SQLite FTS5 + 向量库 POC + rerank |
| 查询审计与反馈 | 记录搜索、点击、反馈和使用分析 | `RetrievalLog`、`AnswerFeedback`、评审质量分析 |

### 6.3 建议新增或预留的领域对象

| 对象 | 作用 | MVP 处理方式 |
| --- | --- | --- |
| `KnowledgeSource` | 一个文件上传、目录、飞书空间、GitHub repo 等来源 | P0 先支持手工上传和团队资料 |
| `Connector` | 知识源连接器类型和配置模板 | P2 再做外部连接器 POC |
| `ConnectorCredential` | 凭证和授权记录 | 真实凭证必须进 `runtime/config` 或加密存储 |
| `SyncJob` | 同步任务实例 | P0 可先记录 upload/index 状态 |
| `IndexAttempt` | 某次索引尝试 | 记录成功、失败、耗时、错误 |
| `KnowledgeDocument` | 被解析后的文档元数据 | P0 必做 |
| `KnowledgeChunk` | 检索切块 | P0/P1 必做 |
| `DocumentSet` | 资料集合 | 可映射为团队资料库、项目资料包、评审规则集 |
| `AccessPolicy` | 文档/集合级访问策略 | MVP 粗粒度，后续细化 |
| `RetrievalLog` | 检索请求和命中记录 | P1 建议落地 |

### 6.4 不建议照搬

1. 不建议近期直接引入 Onyx 的完整基础设施栈。它适合企业级知识平台，但本项目近期目标是单机内网、低运维、runtime 数据隔离。
2. 不建议把产品重心改成通用企业搜索。知识库是需求评审的底座，不是唯一主产品。
3. 不建议第一版就做完整权限同步。先做系统内权限过滤，再挑一个连接器做权限同步 POC。
4. 不建议一开始追求连接器数量。连接器越多，凭证、权限、同步失败、审计和数据泄露风险越高。

### 6.5 对本项目的落地建议

近期最应该吸收五件事：

1. 团队资料库的最小模型应包含 `source -> document -> chunk -> index_attempt`。
2. 每次检索必须带上 `workspace_id`、`project_id`、`user_id` 和权限过滤条件。
3. 资料集合要能绑定到 Agent，例如“某团队 Agent 只能查该团队资料集”。
4. 检索结果必须返回来源引用，审查结论必须能追溯到 chunk/document/source。
5. 连接器凭证和索引数据都属于运行时数据，不进入 git。

## 7. 汇总设计建议

### 7.1 统一领域模型建议

建议把四个项目的借鉴点收敛成以下领域对象，不要分别按项目复制概念：

| 领域 | 建议对象 | 来源启发 | 说明 |
| --- | --- | --- | --- |
| 空间权限 | `Workspace`、`WorkspaceMember`、`ResourceACL`、`WorkspacePermission`、`UserToolPrivilege` | open-cowork、odysseus、Onyx | 统一管理个人/团队/项目空间的可见性、动作权限和工具权限 |
| 工作台体验 | `WorkspaceSurfaceConfig`、`ResearchRun`、`ResearchArtifact` | odysseus | 支撑团队空间内的多模块入口、外部调研与可视化产物 |
| 知识底座 | `KnowledgeSource`、`KnowledgeDocument`、`KnowledgeChunk`、`DocumentSet` | Onyx、odysseus | 支撑团队资料库、项目资料包、审查规则库与轻量记忆层 |
| 同步索引 | `Connector`、`ConnectorCredential`、`SyncJob`、`IndexAttempt` | Onyx | 支撑外部知识源导入和索引状态 |
| Agent 运行 | `AgentProfile`、`AgentRun`、`AgentStep`、`ToolCallTrace` | workflow-cookbook、open-cowork、odysseus | 支撑个人 Agent、团队 Agent、审查 Agent 的可观测执行 |
| 工作流 | `WorkflowTemplate`、`WorkflowRun`、`WorkflowPhase`、`WorkflowArtifact` | workflow-cookbook、odysseus | 支撑多阶段审查、讲解产物和评审会包 |
| 人在环 | `AgentApprovalRequest`、`ReviewParticipant`、`ReviewDecision` | open-cowork、workflow-cookbook、odysseus | 支撑审核人控制权限、关键动作确认和归档 |
| 工具能力 | `SkillPackage`、`ConnectorConfig`、`ActionPolicy`、`MCPServerConfig` | open-cowork、odysseus、Onyx | 支撑 Skills/MCP/Actions 的管理、白名单和审计 |

### 7.2 MVP 不应过度设计的边界

MVP 可以做粗粒度，但不能缺少方向上的锚点：

| 能力 | MVP 需要做 | MVP 不需要做 |
| --- | --- | --- |
| 空间权限 | owner/admin/member/viewer 或更少档位；个人资料默认私有 | 完整 ABAC、跨系统权限同步 |
| 工作台/协作 UI | 团队空间统一入口、资料/审查/调研等少量模块并列 | 邮件/日历/笔记/任务全家桶 |
| 团队资料库 | 上传、解析、chunk、FTS、引用来源 | 大规模连接器市场、复杂 ETL |
| Agent 执行 | 记录 run/step/tool trace；关键动作确认 | VM 沙盒、远程桌面自动化 |
| 工作流 | 内置阶段模板、结构化输出、局部失败可重跑 | 用户自定义 DSL、通用流程画布 |
| 检索 | SQLite FTS5 + 权限过滤 + 来源引用 | 立即上完整 Onyx 栈或大型向量集群 |
| 外部调研 | 多步采集 + 来源归档 + 报告产物 | 通用 Deep Research 平台 |
| MCP/工具 | 工具白名单、少量内置工具、管理员分级权限 | MCP 市场、开放任意外部工具 |

### 7.3 分阶段吸收路线

| 阶段 | 建议吸收内容 | 对应项目 |
| --- | --- | --- |
| P0：团队资料库 MVP | `KnowledgeSource`、`KnowledgeDocument`、`KnowledgeChunk`、FTS、来源引用、空间粗权限、团队空间统一工作台入口 | Onyx、odysseus |
| P1：空间与审查权限 | `WorkspaceMember`、`ResourceACL`、`ReviewParticipant`、关键动作确认、管理员高权限工具分级 | Onyx、open-cowork、odysseus |
| P2：Agent 运行观测 | `AgentRun`、`AgentStep`、`ToolCallTrace`、预算、局部重试 | workflow-cookbook、open-cowork、odysseus |
| P2：RAG/连接器 POC | LanceDB/Milvus Lite/Chroma POC、飞书/Notion/GitHub 连接器样板 | Onyx、odysseus |
| P3：工作流模板库 | `WorkflowTemplate`、阶段化审查、多角色并行、Schema 校验 | workflow-cookbook |
| P3：外部调研与讲解产物 | `ResearchRun`、来源清单、可视化报告、讲解稿草案 | odysseus、workflow-cookbook |
| P3：人在环协作 | 审批面板、通知通道、Agent 待确认回复 | open-cowork、odysseus |
| P4+：企业级扩展 | 权限同步、服务化向量库、MCP 管理台、容器/VM 沙盒、质量评估 | Onyx、open-cowork、workflow-cookbook |

## 8. 对主规划文档的建议补充段落

后续更新 `需求业务工作流的cowork版工具设计规划-20260604.md` 时，建议新增一节：

> 外部 cowork / workflow / knowledge 项目的分层借鉴  
> 本项目不直接复制任何单一开源项目。`workflow-cookbook` 作为确定性工作流设计参考，补强 WorkflowTemplate、AgentRun、阶段预算、Schema 输出和对抗验证；`open-cowork` 作为本地 Agent 产品化参考，补强工作区授权、Skills/MCP 管理、Trace、工具审计和人在环审批；`odysseus` 作为 Web 化自托管 AI 工作台参考，补强统一 workspace、Deep Research 产物流、管理员高权限工具治理和默认安全部署约束；`onyx` 作为企业知识底座参考，补强连接器、Document Set、权限感知检索、索引状态、Agent knowledge scope 和使用分析。近期以轻量模型落地，保持 FastAPI + SQLite + runtime 数据隔离；中后期再评估服务化向量库、MCP 管理台、外部权限同步和沙盒执行环境。
