# Claude Cowork 对需求初审工作流的迭代启示

> 日期：2026-06-02
> 资料来源：Claude Cowork 官方博客、产品页与帮助中心资料
> 目标：从 Claude Cowork 的设计思路和功能组合中，提炼可借鉴到当前“AI 产品需求初审”平台的迭代方向。

## 一、结论摘要

Claude Cowork 的核心启示不是“把聊天窗口做强”，而是把 AI 产品从“回答问题”推进到“接收目标、拆解任务、调度工具、交付产物、接受治理”的工作流系统。

对当前需求初审平台而言，最值得借鉴的方向有六个：

1. 把“审查按钮”升级为“可委托的评审任务”：用户描述目标，系统生成执行计划、确认输入范围、运行流水线并沉淀结果。
2. 把“项目”升级为长期工作空间：项目内有需求文档、规则文档、历史评审、团队记忆、计划任务和专属指令。
3. 把现有 Skills 升级为“评审插件包”：每个插件包组合技能、连接器、子 Agent、输出模板和权限策略。
4. 把流水线从线性模式扩展为“多角色并行评审 + 汇总裁决”：产品、技术、测试、交付、合规等视角可并行产出意见，再由总评审器合并冲突。
5. 把“结果报告”升级为专业交付件：除 Markdown 报告外，支持评审纪要、问题清单、PRD 修订稿、会议包、看板数据等可继续协作的产物。
6. 把“能运行”升级为“可治理”：管理员应能看到团队采用情况、技能调用、成本耗时、文件访问、人工确认、失败率和质量趋势。

当前系统已经具备很好的基础：项目化评审、Review Context、SkillRunner、任务状态、SSE 进度、运行时数据隔离、后台模型和 Prompt 管理。这些能力与 Cowork 的项目、插件、任务、观测、权限方向高度同构。后续迭代重点应放在“任务委托体验、插件化治理、项目记忆、计划任务、观测面板和专业交付件”上。

## 二、Claude Cowork 的设计拆解

### 2.1 从 Chat 到 Task：让 AI 执行完整工作

Cowork 与普通 Chat 的关键区别是：用户不再只问一个问题，而是交代目标、范围和节奏，让 AI 自己拆解步骤、执行工具、处理文件，并返回完成品。官方资料反复强调 Cowork 适合复杂、多步骤、需要文件访问和较长执行时间的知识工作。

对我们的启示：

- 需求初审不应只是“上传文档后跑一个模式”，而应包装成“审查任务委托”。
- 用户创建任务时，系统先展示计划：本次会读哪些文档、使用哪些规则、调用哪些 Skills、预计生成哪些产物。
- 任务执行中允许用户中途补充方向，而不是只能等待流水线结束。
- 任务结束后不只展示模型回复，而要交付结构化产物和下一步动作。

### 2.2 项目化工作空间：文件、指令、上下文和记忆同域管理

Cowork Projects 将相关任务组织到独立工作空间中，每个项目有自己的文件、链接、指令、计划任务和记忆。记忆限定在项目内，不跨项目泄漏。

对我们的启示：

- 当前 `Review Project + Review Context` 已经接近这个方向，但还偏“审查执行容器”。
- 后续可升级为“需求项目工作空间”，让每个项目长期保存：
  - 待审需求文档
  - 历史版本和历史评审意见
  - 团队规则、模板、术语表、评分量规
  - 项目专属指令，例如输出格式、关注风险、业务边界
  - 项目记忆，例如已确认的边界、反复出现的问题、历史争议点
  - 计划任务，例如每周自动检查新增文档或重新汇总缺口

关键设计点：项目记忆必须版本化、可查看、可删除、可回滚，不能变成不可解释的隐式上下文。

### 2.3 插件组合：Skills、连接器和子 Agent 打包成团队能力

Cowork 的插件把 Skills、连接器和子 Agent 打包，让 Claude 从第一轮对话就具备某个团队或角色的专业能力。组织可以通过插件市场分发团队级插件，并限制不同群组可用的插件。

对我们的启示：

- 当前仓库已有多个本地 Skills 和 SkillRunner，可以进一步产品化为“评审插件包”。
- 一个评审插件包不只是 Prompt，而应包含：
  - 技能清单：例如文档转换、分类、逐篇分析、体系 Review、报告生成
  - 输入输出 Schema：保证结果可落库、可比较、可回归
  - 连接器声明：例如本地文件库、飞书导入、历史需求库、问题看板
  - 子 Agent 角色：例如产品审查、技术审查、测试审查、交付审查
  - 输出模板：例如初审报告、会议纪要、PRD 修订建议、问题清单
  - 权限策略：谁可安装、谁可运行、是否允许自动写入外部系统

这能把“团队评审经验”从散落的 Prompt 和页面按钮，收敛成可分发、可升级、可审计的能力包。

### 2.4 长任务与并行工作：从线性流水线到多角色协作

Cowork 会根据任务复杂度拆成子任务，并在合适时协调多个工作流并行处理。我们的当前流水线以 1→2→3→4→5→6 为主，稳定但偏线性。

可借鉴的改造：

- 文档转换、分类和逐篇分析可以保持确定性串行。
- 逐篇分析之后，体系 Review 可拆成多角色并行：
  - 产品价值评审
  - 技术可行性评审
  - 交互与体验评审
  - 测试与验收评审
  - 数据与指标评审
  - 风险与合规评审
- 汇总阶段由“总评审器”处理冲突、去重、排序和证据引用。
- 前端进度从“步骤列表”升级为“任务拓扑图”，显示每个角色视角的状态、耗时、失败和重试。

这类改造能提升复杂需求集的吞吐，也让评审意见更贴近真实评审会的多角色协作方式。

### 2.5 计划任务：把一次性初审变成持续治理

Cowork 支持计划任务，用户可以定义一次说明，让 AI 按固定节奏或按需运行，并查看历史运行结果。

对需求初审平台可落地为：

- 每周自动汇总项目新增需求文档，生成“本周需求质量简报”。
- 当项目规则文档更新后，提示是否对未关闭需求重新审查。
- 定期追踪历史“边界外问题”是否在后续版本中收敛。
- 每月生成团队维度的需求质量趋势和典型问题。
- 对长期未处理的评审问题生成提醒或待办。

这会把平台从“文档提交前自检工具”推进为“需求质量运营系统”。

### 2.6 人在环与权限控制：允许自动化，但把关键动作收口

Cowork 的体验强调：用户选择文件和连接器权限；关键动作前展示计划并请求确认；永久删除等高风险动作必须额外确认；管理员可做角色访问、预算、连接器动作限制和观测配置。

对当前平台的直接启示：

- 审查开始前展示“执行计划确认”：
  - 输入文档范围
  - 使用的评审上下文版本
  - 使用的模型与思考级别
  - 将要调用的 Skills
  - 是否会写入公共库、外部看板或生成草稿
- 高风险动作单独确认：
  - 覆盖旧报告
  - 删除文档
  - 批量重跑并替换历史结果
  - 将问题写入外部协作系统
- 管理后台增加能力开关：
  - 哪些角色可运行 full/draft 等高成本模式
  - 哪些角色可管理公共文件库和团队规则
  - 哪些连接器允许读、写或只读
  - 每用户或每团队的运行额度

这与本项目“内网部署、数据不外溢、运行时数据隔离”的目标一致。

### 2.7 观测与采用分析：管理者要知道哪些工作流真的有价值

Cowork 的企业化重点包括使用分析和 OpenTelemetry 观测：管理员可以跟踪会话、活跃用户、技能和连接器调用、工具动作、文件访问、人工批准状态、模型请求、耗时、成本和错误等。

当前平台已有审计日志、会话日志、任务状态和后台统计，但还没有形成“需求评审运营仪表盘”。建议后续补充：

- 工作流采用：
  - 各审查模式运行次数
  - 活跃用户和活跃项目
  - 人均审查文档数
  - full/draft/insight 的采用趋势
- 技能质量：
  - 每个 Skill 调用次数、耗时、失败率、重试次数
  - Schema 修复次数
  - 缓存命中率
  - 各模型在同一 Skill 上的失败率和成本
- 需求质量：
  - 平均评分趋势
  - 高频缺失章节
  - 高频边界外问题
  - 问题关闭和版本收敛情况
- 安全治理：
  - 文件访问记录
  - 连接器读写动作
  - 人工确认/拒绝记录
  - 高风险操作审计

这类数据可以反向指导 Prompt、Skill 和团队规范的持续优化。

## 三、与当前系统能力的映射

| Cowork 设计 | 当前系统已有能力 | 差距 | 可迭代方向 |
| --- | --- | --- | --- |
| 目标驱动任务 | 评审模式按钮、ReviewTask、SSE 进度 | 用户仍需理解模式差异，缺少执行计划确认 | 增加“创建评审任务向导”和“执行计划预览” |
| 项目工作空间 | ReviewProject、ReviewContext、文档管理 | 项目记忆、计划任务、长期规则沉淀不足 | 项目记忆、规则版本、计划任务、历史问题闭环 |
| 插件包 | skills/ 本地技能、SkillRunner | Skills 还偏工程目录，不是可配置产品能力 | 评审插件包、插件版本、启停、权限和回归样例 |
| 子 Agent 协作 | system-review 多维 Prompt | 多维度仍由单个 Skill 统一处理 | 多角色并行评审 + 汇总裁决 |
| 专业产物 | Markdown 报告、PRD 草稿 | 输出形态偏报告，协作产物不足 | 问题清单、评审纪要、修订建议、看板导出 |
| 计划任务 | 暂无稳定能力 | 不能自动周期复查 | 周报、重审提醒、缺口追踪、质量趋势 |
| 人在环审批 | 任务取消、失败保留中间结果 | 缺少高风险动作确认和计划确认 | 执行前确认、写入前确认、覆盖/删除确认 |
| 企业治理 | 用户、模型、Prompt、技能管理，审计日志 | 缺少配额、角色级能力、采用分析 | RBAC 细化、模式额度、技能调用统计、观测面板 |

## 四、建议的产品迭代方向

### 方向 1：评审任务向导

把“选择模式 + 点击开始”改成“目标驱动的任务创建”：

1. 用户选择目标：快速自检、提交前评审、评审会准备、生成下一版 PRD、PM 能力复盘。
2. 系统推荐模式和输入范围。
3. 系统展示执行计划。
4. 用户确认后执行。
5. 任务完成后展示交付件和下一步建议。

短期可先做为前端向导，不改底层 SkillRunner。

### 方向 2：项目工作空间增强

在当前 Review Project 基础上补齐长期协作能力：

- 项目规则版本：记录每次审查使用的规则版本。
- 项目记忆：沉淀已确认边界、历史争议、常见问题和团队偏好。
- 项目资料分层：区分待审文档、历史文档、规则文档、临时资料、外部链接。
- 项目级计划任务：定期重审、周报、缺口追踪。

这与 V0.2.6 公共文件库和飞书导入方向可以合并规划：公共文件库提供资料资产，项目工作空间负责引用和上下文编排。

### 方向 3：评审插件包

将现有 Skills 产品化为可治理能力：

- 插件包元数据：名称、版本、适用场景、输入要求、输出产物、风险等级。
- 插件包权限：哪些角色可运行，是否允许外部连接器，是否允许写操作。
- 插件包回归：每个插件包绑定样例文档和期望结构，升级前自动验证。
- 插件包市场：管理员发布团队标准评审包，普通用户安装或使用。

第一版不需要真正做插件市场，可以先在后台“技能管理”增加分组、版本、启停和样例验证入口。

### 方向 4：多角色并行评审

把体系 Review 拆为可并行、可追踪、可单独重跑的角色视角：

- 产品价值 Agent
- 技术可行性 Agent
- 测试验收 Agent
- 交付风险 Agent
- 数据指标 Agent
- 规范合规 Agent
- 总评审 Agent

每个角色输出统一 Schema：发现、证据、严重级别、建议动作、关联文档、是否阻断。总评审 Agent 负责合并、排序、消除重复和生成最终建议。

### 方向 5：专业交付件矩阵

从“报告”扩展为“交付件集合”：

| 场景 | 交付件 |
| --- | --- |
| PM 自检 | 问题清单、缺失章节、修订建议 |
| 技术评审 | 风险清单、边界问题、待确认问题 |
| 评审会 | 会议议程、争议点、决策项、行动项 |
| 下一版规划 | 缺口矩阵、优先级列表、PRD 草稿 |
| 团队管理 | 质量趋势、PM 评分卡、规范命中率 |

这能让平台更贴近日常协作，而不是只输出一份“看完即结束”的报告。

### 方向 6：评审运营与治理面板

后台新增“评审运营”视图：

- 用量：用户、项目、任务、文档、模式分布。
- 效率：平均耗时、排队时间、失败率、重试率。
- 成本：模型调用、Token、不同模式消耗。
- 质量：评分趋势、高频问题、缺失章节、缺口收敛。
- 安全：文件访问、外部导入、人工确认、高风险操作。

这能帮助团队判断哪些功能真正提高了评审质量，而不是只看功能是否上线。

## 五、建议实施顺序

### P0：不改底层的体验增强

- 新增评审任务向导和执行计划预览。
- 将审查模式改为业务目标文案。
- 任务完成页显示“交付件集合”和“下一步建议”。
- 后台统计补充模式次数、平均耗时、失败率。

### P1：项目工作空间与公共资料能力

- 落地公共文件库和项目引用关系。
- 为项目增加规则版本、资料分层和项目指令。
- 增加项目记忆的显式查看、编辑、删除能力。
- 将飞书导入作为连接器能力接入，不直接写死在审查流程中。

### P2：评审插件包与多角色评审

- 把 Skills 分组为评审插件包。
- 增加插件包版本、启停、样例回归。
- 将 system-review 拆成多角色并行评审。
- 增加汇总裁决和冲突合并。

### P3：计划任务与治理闭环

- 增加项目级计划任务。
- 支持定期质量简报、缺口追踪和规则更新后的重审提醒。
- 增加角色级能力控制、配额和成本视图。
- 增加文件访问、连接器动作和人工确认审计。

## 六、当前项目可立即吸收的设计原则

1. 任何 AI 审查都应先说明计划，再执行。
2. 任何长期项目都应有显式上下文、显式记忆和可追溯规则版本。
3. 任何团队经验都应沉淀为可升级的能力包，而不是散落在 Prompt 中。
4. 任何自动化写入、覆盖、删除、外部同步都应有人工确认和审计。
5. 任何复杂评审都应支持多视角并行，而不是让一个 Prompt 承担所有角色。
6. 任何上线的工作流都应能被观测：谁在用、用在哪、效果如何、成本多少、失败在哪里。
