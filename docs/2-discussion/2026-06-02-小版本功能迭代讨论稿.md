# V0.2.6 小版本功能迭代讨论稿

日期：2026-06-02

## 背景

本次讨论聚焦 5 个小版本能力：

1. 管理后台为每个模型配置“思考级别”。
2. 管理后台导入公共文件库，供所有用户复用。
3. 调研飞书 CLI / 飞书开放平台，评估是否能把飞书链接转成文档 Markdown 并接入系统。
4. 修复聊天消息 Markdown 显示和 Mermaid 流程图渲染问题。
5. 将页面品牌、Logo、favicon、标题和主题色改为 runtime 本地个性化配置。

当前 repo 的相关基线：

- 模型配置在 `model_configs`，目前包含 `max_tokens`、`temperature`、启用状态、测速状态等字段。
- 普通对话调用走 `src/app/services/llm.py::stream_chat()`，审查 pipeline 的结构化调用走 `src/app/services/retry.py::structured_chat()`。
- 审查文档在 `review_documents`，按项目归属，上传文件保存在 runtime 下，不适合直接作为全站公共库。
- 审查页已有“项目上下文 / 历史文档 / 需求规范 / 团队指导意见”区域，可承接公共库选择入口。
- 聊天页已引入 `DOMPurify`、`marked`、`mermaid`，但依赖 CDN 且 fallback Markdown 解析能力较弱；截图中 `###`、`*` 原样显示，说明运行时 Markdown 渲染链路没有稳定生效。
- 登录页、工作区左上角、favicon、页面 title 和主题色当前仍有硬编码项；通用代码覆盖旧项目时，容易把旧项目的品牌展示替换成通用版。

## 需求 1：模型思考级别

### 调研结论

不同模型供应商对“思考”的支持方式不一致：

- OpenAI 新推理模型通常通过 reasoning / effort 一类参数控制推理强度。
- DeepSeek reasoning 模型会返回推理内容字段，但不同模型是否支持关闭或调档，需要按模型能力判断。
- Qwen 等模型存在 enable thinking / thinking budget 这类开关或预算参数。
- 许多 OpenAI-compatible 网关不会接受未知字段；直接把 `reasoning_effort` 固定塞进所有请求，可能导致 HTTP 400。

因此，“思考级别”不应只做一个前端枚举字段，而应抽象成模型配置能力，再由 provider adapter 转成具体 payload。

### 推荐方案

推荐采用“三档运行时选择 + 后端适配层”的设计：

- `off`：关闭思考，前端显示为“关”。
- `low`：低思考。
- `high`：高思考。

数据库建议在 `model_configs` 增加：

- `thinking_level`：默认思考级别，枚举 `off|low|high`。
- `thinking_supported`：是否支持思考配置。
- `thinking_adapter`：适配方式，建议枚举 `none|openai_reasoning|deepseek_reasoner|qwen_thinking|custom_json`。
- `thinking_payload`：可选 JSON，用于网关私有参数映射。

调用层建议新增统一方法：

```python
build_llm_payload(base_payload, model_cfg)
```

它负责把抽象配置转换成供应商参数。对不支持的模型不传任何思考字段，避免破坏兼容性。

### 前端改造建议

智能对话页和需求审查页的模型选择器右侧增加“思考级别”下拉菜单。下拉选项固定为：

- `关`
- `low`
- `high`

交互规则：

- 当前模型不支持思考配置时，下拉置灰或显示“不可用”。
- 用户切换思考级别后，后续智能对话请求和需求审查启动请求都使用新的思考级别。
- 该选择应作为运行时覆盖值随请求发送，而不是只修改后台模型默认配置。
- 前端可用 `localStorage` 记住最近一次选择；切换模型时若新模型不可用，则自动回退为“关”。

管理后台模型列表仍保留“思考级别/支持状态”列，用于显示该模型是否支持思考配置：

- 不支持：显示“不可用”，置灰。
- 支持：显示默认级别（关/low/high）。

模型配置弹窗中增加：

- “支持思考”开关。
- “默认思考级别”选择（关/low/high）。
- “适配类型”选择。
- 高级区域：自定义 payload JSON，默认隐藏。

第一版采用“管理员配置模型是否支持 + 用户在模型选择器旁运行时选择级别”的组合方式。管理员负责声明模型能力和适配方式，用户只选择关、low、high 三档。

### 风险

- 各模型参数名不同，必须按 provider 分支处理。
- 有些模型思考会显著增加延迟和 token 成本，测速结果应记录思考级别。
- 推理过程展示（`reasoning_content`）已在 V0.2.6 DEV-008 中同步交付：流式渲染 `.msg-reasoning`、`StreamChunk.reasoning_content` 字段、`log_llm_session` 记录，与”调档”功能组合提供完整的思考级别体验。
- 运行时覆盖值需要同时覆盖聊天流式请求和审查 pipeline 请求，否则同一页面上看起来选择了思考级别，但审查任务实际没有使用。

## 需求 2：后台公共文件库

### 问题定义

目标是管理员统一导入一批公共文档，所有访问网站的人都可以在审查项目或对话中复用，不需要每个用户重复上传。

这里要区分两种使用方式：

1. **引用**：公共文档仍属于公共库，用户项目只保存引用关系。
2. **拷贝**：用户选择公共文档后，在自己的项目下生成一份项目文档。

推荐第一版采用“引用为主，必要时落项目快照”的方案：公共库负责存储和转换，项目只保存引用，启动审查时把公共文档展开到 pipeline 输入。

### 数据模型建议

新增公共库表：

```text
library_documents
- id
- title
- source_type: upload|url|feishu
- original_filename
- file_path
- md_path
- content_hash
- document_type: requirement|historical|spec|guidance|other
- visibility: public
- tags
- status: uploaded|converted|failed|archived
- created_by
- created_at
- updated_at
```

新增项目引用表：

```text
project_library_documents
- id
- project_id
- library_document_id
- usage_type: review_target|historical|spec|guidance
- added_by
- created_at
```

为什么不直接复用 `review_documents`：

- `review_documents.project_id` 必填，天然是项目私有。
- 公共库需要全站可见、可标签检索、可统一更新。
- 直接塞一个“公共项目”会让权限、删除、审查范围和缓存逻辑变复杂。

### API 建议

管理后台：

- `GET /api/admin/library-documents`
- `POST /api/admin/library-documents/upload`
- `POST /api/admin/library-documents/url`
- `DELETE /api/admin/library-documents/{id}`
- `POST /api/admin/library-documents/{id}/convert`

用户侧：

- `GET /api/library-documents?type=&q=&tag=`
- `POST /api/review/projects/{project_id}/library-documents`
- `DELETE /api/review/projects/{project_id}/library-documents/{library_doc_id}`

审查启动时：

- `document_ids` 仍表示项目私有文档。
- 新增 `library_document_ids` 或由项目引用关系自动展开。
- `step_details` 记录公共文档引用，保证缓存和报告可追溯。

### 前端改造建议

管理后台新增“公共文件库”tab：

- 上传文档。
- URL/飞书链接导入。
- 文档类型、标签、状态、转换结果。
- 重新转换、归档、删除。

审查页左侧“添加文档”区域增加两个入口：

- “上传本地文档”
- “从公共库选择”

项目上下文区域的“历史文档”也增加“从公共库引用”，公共历史资料不必每个项目重复导入。

### 权限建议

- 管理员可以新增、删除、归档公共库文档。
- 普通用户只能引用公共库文档，不能修改公共库原件。
- 公共库文档被项目引用后，删除前应提示已有引用数量；默认归档而不是硬删。

### 风险

- 公共文档更新后，历史审查报告应保持可追溯。建议报告记录 `content_hash`。
- 公共库文档可能包含敏感信息，必须有明确管理员入口和审计日志。
- 如果公共库文档参与缓存，缓存 key 必须包含 `content_hash`，不能只用文档 ID。

## 需求 3：飞书链接导入 / 飞书 CLI 集成

### 调研结论

可选路径有三类：

1. **飞书开放平台 API**  
   官方云文档接口支持获取文档内容，返回格式可用于 Markdown 化。需要企业自建应用、应用凭证、权限申请和文档授权。

2. **larksuite/cli**  
   官方 GitHub 仓库存在 Lark/Feishu CLI 工具，但它更偏开放平台开发和资源操作，不一定直接满足“用户贴链接 -> 导出 Markdown”的产品链路。

3. **第三方/自建导入器**  
   使用飞书开放平台 SDK 或 HTTP API，自建一个 `FeishuImportService`，把链接解析、token 获取、文档读取、Markdown 保存统一收口。

推荐第一版不要把“CLI”作为核心运行依赖，而是采用“服务端 API 集成优先，CLI 作为调研/运维辅助”的方案。

### 前置条件

需要准备：

- 企业内创建飞书自建应用。
- 获取 `app_id`、`app_secret`。
- 申请云文档读取相关权限。
- 明确使用 tenant access token 还是 user access token。
- 目标文档必须授权给应用，或由有权限的用户完成 OAuth 授权。
- 明确支持的链接类型：docx/doc/sheet/wiki/bitable 是否都要支持。

### 推荐业务流程

第一版先做“后台导入飞书文档到公共库”：

1. 管理员在公共文件库输入飞书链接。
2. 后端解析链接，提取文档 token 和文档类型。
3. 后端用飞书应用凭证读取文档内容。
4. 转换成 Markdown，保存到 `runtime/data/library_documents/...`。
5. 写入 `library_documents`，来源标记为 `feishu`。
6. 用户在项目里从公共库引用该文档。

第二版再做“普通用户在聊天或审查页粘贴飞书链接”：

- 如果系统使用应用级 token 且文档已授权，直接读取。
- 如果文档未授权，提示用户联系管理员导入，或走 OAuth。

### 为什么不建议一开始让普通用户直接贴任意飞书链接

- 权限问题最复杂：用户能看，不代表应用能看。
- 飞书文档链接类型多，解析规则不完全一致。
- 文档里可能有图片、表格、嵌套块，需要稳定转换链路。
- 如果没有 OAuth，系统无法代表用户读取其私有文档。

### 前端改造建议

管理后台公共库新增“飞书导入”：

- 输入飞书链接。
- 显示解析结果：文档类型、标题、权限状态。
- 导入后展示 Markdown 转换状态。

用户侧公共库选择器中显示来源：

- 本地上传
- URL
- 飞书

如果用户在对话框粘贴飞书链接，第一版只做识别和提示：

- “这是飞书链接，请到公共文件库导入后使用。”

等 OAuth 或应用授权跑通后，再开放自动导入。

## 需求 4：Markdown 显示与 Mermaid 流程图渲染

### 问题现象

聊天消息中 AI 回复包含 Markdown 时，页面可能直接显示原始语法，例如：

- `### 核心业务逻辑梳理` 没有渲染为标题。
- `* T1时刻` 没有渲染为列表。
- Mermaid 代码块没有渲染为流程图。

当前代码已有 `_renderMarkdown()` 和 `_renderMermaidCharts()`，但页面实际表现说明渲染链路不稳定。主要风险点有三类：

1. 前端依赖通过 CDN 加载，内网或离线部署时可能拿不到 `marked`、`DOMPurify`、`mermaid`。
2. fallback Markdown 只处理少量语法，不支持标题、列表和 Mermaid 容器。
3. 流式输出过程中每个 chunk 都会触发 Mermaid 渲染，代码块还未闭合时可能提前失败。

### 推荐方案

推荐采用“本地化依赖 + 流式结束后 Mermaid 渲染 + fallback 增强”的方案。

补充调研结论：

- 主方案继续沿用 `marked + DOMPurify + mermaid`。当前代码已经采用这套主流组合，P2 的问题不在于选型错误，而在于依赖加载、fallback、流式渲染时机和错误态没有补完整。
- `markdown-it + DOMPurify + mermaid` 作为后续增强备选。它插件生态更强，但不能直接解决当前内网/CDN、流式 Mermaid 误触发和错误回显问题，本阶段不建议为了 P2 大换栈。
- fallback 只承担保底显示能力，不做重型自研 Markdown 解析器。目标是库不可用时仍能阅读标题、列表、代码块和 Mermaid 源码，而不是完整复刻 `marked`。
- Mermaid 采用编程式集成：`startOnLoad: false`，完整消息结束后再 `parse/render`；语法错误时进入失败态并保留源码。

前端静态资源：

- 将 `DOMPurify`、`marked`、`mermaid` 放到 `src/static/vendor/`，由本地静态文件加载。
- 本地 vendor 作为主路径，CDN 只作为可选兜底；发布验收不能依赖公网 CDN。
- 记录版本、来源和许可证，避免后续升级时不可追溯。

Markdown 渲染：

- `marked + DOMPurify` 仍作为主路径，`marked` 负责 Markdown 转 HTML，`DOMPurify` 负责输出清洗。
- fallback 至少支持标题、段落、无序/有序列表、粗体、斜体、行内代码、fenced code block。
- fallback 也要识别 ```` ```mermaid ````，生成 `.mermaid-chart[data-mermaid]` 容器。
- fallback 不继续堆复杂正则特性，避免安全和维护风险扩大。
- fallback 复杂度需要可审查：代码规模原则上不超过主渲染逻辑的 1/3；超过时优先删减 fallback 能力，而不是继续扩展正则规则。

Mermaid 渲染：

- 流式输出过程中继续更新 Markdown 文本，但不急着渲染 Mermaid。
- 收到 `[DONE]`、请求正常结束、SSE `error/close/finally` 兜底时，再对完整消息统一渲染 Mermaid，且同一条消息只触发一次最终渲染。
- 增加 idle debounce 兜底：最后一块 chunk 到达后若约 2 秒没有新 chunk，也触发一次最终 Mermaid 渲染；仍只处理已闭合的 Mermaid fenced block。
- Mermaid 初始化使用 `startOnLoad: false`，渲染前先做 `parse`，通过后再 `render`。
- 只对已经闭合的 Mermaid fenced block 创建图表容器。
- Mermaid 渲染失败时保留原始源码或提供“查看源码”区域，不要直接移除诊断信息。
- 明确 DOMPurify 与 Mermaid 的执行顺序：先由 `marked -> DOMPurify` 清洗 Markdown 生成的普通 HTML 和 Mermaid 容器，再由 Mermaid 在容器内生成 SVG；不要在 Mermaid SVG 生成后再套用仅 HTML profile 的 DOMPurify 清洗。如确需清洗 SVG，必须显式允许 SVG profile，避免图表被静默破坏。

### 前端改造建议

聊天页：

- 在 `_renderMarkdown(text)` 中拆出 `_renderMarkdownWithLibraries()` 和 `_renderMarkdownFallback()`，便于测试。
- 统一聊天页和审查结果页的 Markdown/Mermaid 渲染策略，避免两个页面各自维护不同 fallback。
- 在 `sendMessage()` 中把 `this._queueMermaidRender(contentEl)` 从每个 chunk 移到流结束后。
- 在 SSE 异常关闭和 idle debounce 场景下补最终渲染兜底，避免“流式中断后 Mermaid 不出现，刷新历史消息才出现”。
- 历史消息 `_appendMessage()` 仍可立即触发 Mermaid 渲染，因为历史消息已是完整文本。
- 给 Mermaid 容器增加加载态和失败态样式；该样式改造与 Mermaid 渲染时机作为同一交付单元处理。

测试建议：

- 前端契约测试覆盖 Markdown 标题、列表、代码块、Mermaid 容器。
- 增加“CDN 库不存在时 fallback 仍能渲染标题/列表”的测试。
- 增加“流式过程中不提前渲染 Mermaid，结束后才渲染”的测试。
- 增加“SSE 异常中断或 2 秒无新 chunk 时仍触发最终 Mermaid 渲染”的测试。
- 增加“DOMPurify 不破坏 Mermaid SVG 输出”的测试或浏览器验证。
- 增加“Mermaid 语法错误时展示错误态并保留源码”的测试。
- 增加浏览器真实验证：在 Chrome DevTools Network 面板阻断 `jsdelivr.net` 后，新建对话发送含标题、列表、Mermaid 代码块的消息，确认标题和列表正常渲染，Mermaid 图表在流式结束或异常兜底后出现。

### 风险

- Mermaid 图语法由模型生成，语法错误不可避免，需要保留错误提示和源码。
- SSE 连接异常关闭时可能收不到 `[DONE]`，必须有 `error/close/finally` 和 idle debounce 兜底。
- Mermaid 输出 SVG，DOMPurify 的清洗顺序和 profile 配置不当会导致图表静默失败，需要在实现和测试中显式覆盖。
- 本地化第三方前端库要注意版本和许可证记录。
- 如果所有回答都强制 `breaks: true`，长段落可能换行过密；可针对 AI 消息保留 GFM，但控制段落和列表样式。

## 需求 5：品牌与本地个性化配置

### 问题定义

当前登录页 Logo、工作页面左上角 Logo、favicon、页面 title、产品名、登录页说明文案和主题色存在硬编码。由于我们会把新版通用代码覆盖到旧项目中运行，如果这些元素继续写在源码里，升级后旧项目的本地品牌展示会被通用版覆盖。

这个需求建议作为 V0.2.6 的 P3 独立小需求，名称为：

```text
P3 品牌与本地个性化配置
```

目标不是做复杂的多租户主题系统，而是提供一个稳定的“本地 runtime 覆盖”机制，让不同项目在保留 runtime 的情况下，升级通用代码后仍能显示自己的 Logo、标题、favicon 和主题色。

### 推荐方案

采用“代码内默认配置 + runtime 本地覆盖 + runtime 本地资产目录”的设计。

配置文件约定：

```text
runtime/config/ui-branding.yaml
```

资产目录约定：

```text
runtime/assets/branding/
```

配置优先级：

1. `runtime/config/ui-branding.yaml`
2. `src/config.yaml` 或后端默认配置中的通用品牌字段
3. 前端内置兜底默认值

建议配置项：

```yaml
app_title: "AI产品需求初审"
favicon: "favicon.svg"
login_logo: "login-logo.svg"
topbar_logo: "topbar-logo.svg"
login_title: "AI产品需求初审"
login_subtitle: "AI 驱动的需求审查工作流平台"
```

其中 `favicon`、`login_logo`、`topbar_logo` 只允许引用 `runtime/assets/branding/` 下的文件，避免配置任意本机路径或外部 URL 带来安全和迁移风险。

### 后端改造建议

- 新增 `BrandingConfigService`，负责加载默认配置、读取 `runtime/config/ui-branding.yaml`、合并字段、校验资产路径。
- 新增 `GET /api/app/branding`，返回前端需要的品牌文案、主题色和资产 URL。
- 新增安全资产访问路由，例如 `/assets/branding/{path}` 或 `/api/assets/branding/{path}`，只允许读取 `runtime/assets/branding/` 下的文件。
- 配置缺失、字段错误、资产文件不存在时只做字段级回退，不阻断应用启动。
- 对非法路径、资产缺失、配置解析失败写入日志，方便升级排查。

### 前端改造建议

- 应用初始化时加载 `/api/app/branding`，失败时使用默认值。
- 登录页 Logo、标题、副标题、提示文案从配置渲染。
- 工作区左上角 Logo 和产品名从配置渲染。
- `document.title` 和 favicon 根据配置动态更新。
- CSS 主题变量由配置注入，至少覆盖主色、hover 色和强调色。
- 顶栏入口文案可配置，但默认保持当前通用文案。

### 验收标准

核心验收标准：

```text
把新版代码覆盖到旧项目后，只要旧项目保留 runtime/config/ui-branding.yaml 和 runtime/assets/branding/，
页面 Logo、标题、favicon、主题色仍按旧项目显示。
```

补充验收：

- 旧 runtime 没有 branding 配置时，页面使用通用默认品牌且应用正常启动。
- branding 配置中的非法路径不会被读取，前端自动回退默认资产。
- 仓库不提交具体企业 Logo、内部品牌名、内部域名或旧项目资产。
- 自动化测试覆盖配置合并、路径安全、API 返回、前端契约和旧 runtime 兼容场景。

### 风险

- 如果前端在配置返回前就渲染硬编码品牌，页面会短暂闪烁；可先用默认值渲染，再平滑替换。
- favicon 动态替换要处理浏览器缓存，必要时给本地资产 URL 加版本或 mtime 参数。
- 主题色覆盖范围需要控制，第一版只覆盖主色系，不把整个设计系统做成可配置，避免复杂度失控。
- 本地资产路径必须严格限制在 `runtime/assets/branding/`，否则会破坏运行时数据隔离和安全边界。

## 推荐实施顺序

### 阶段 1：模型思考级别

这是最小且边界清晰的功能。建议先做：

- 数据库字段迁移。
- 后端 model config API 扩展。
- LLM payload builder。
- 管理后台模型配置 UI。
- 针对不支持思考字段的模型做“不传字段”保护测试。

### 阶段 2：Markdown 与 Mermaid 渲染修复

这是现有聊天体验问题，改动范围主要在前端静态资源和 `chat.js`/`review.js`，可与 P1 后端能力并行：

- 明确不换栈，继续用 `marked + DOMPurify + mermaid`。
- 本地化 Markdown/Mermaid 依赖。
- 增强最小 fallback Markdown。
- Mermaid 改为完整消息后渲染，并补 SSE 异常中断与 idle debounce 兜底。
- Mermaid 失败时保留源码和错误提示。
- 明确 DOMPurify 与 Mermaid SVG 的清洗顺序，避免图表被清洗破坏。
- 增加前端契约测试。

### 阶段 3：品牌与本地个性化配置

这是新版通用代码覆盖旧项目前需要优先补齐的迁移保护能力：

- 新增 `runtime/config/ui-branding.yaml` 配置约定。
- 新增 `runtime/assets/branding/` 品牌资产目录约定。
- 后端提供 branding 配置读取、路径校验和只读 API。
- 前端用配置驱动登录页、工作区左上角、favicon、title 和主题色。
- 用历史 runtime 验证：旧项目保留本地配置和资产后，覆盖新版代码仍显示旧项目品牌。

### 阶段 4：公共文件库

先做本地上传和公共库引用，不急着接飞书：

- 新表和 repository。
- 后台公共库 tab。
- 用户项目引用公共库文档。
- 审查 pipeline 展开公共库文档。
- 报告记录公共文档来源和 hash。

### 阶段 5：飞书导入

在公共文件库稳定后接入飞书：

- 先支持管理员导入。
- 先支持飞书文档正文 Markdown。
- 暂不承诺所有图片/表格/多维表格完整还原。
- OAuth 和普通用户粘贴链接放到后续版本。

## 需要决策的问题

1. 模型思考级别是否按当前设计固定为 `关 / low / high` 三档，并在智能对话和需求审查中都作为运行时覆盖值生效？
2. 公共文件库里的文档是否允许普通用户上传后“提交给管理员审核入库”？
3. 公共库文档更新后，旧报告是否必须保留旧版本快照？
4. 飞书集成第一版是否只做管理员导入，不做普通用户 OAuth？
5. 飞书导入范围是否先限定为文档正文，图片和表格作为后续增强？
6. Markdown/Mermaid 修复是否作为独立体验修复优先完成，还是并入 V0.2.6 功能迭代一起发布？
7. 是否允许继续保留 CDN 作为备用，还是完全切换到本地 vendor 依赖？默认建议：本地 vendor 为主，CDN 只做可选兜底，发布验收不依赖 CDN。
8. 品牌与本地个性化配置是否按 P3 独立交付？默认建议：是，作为通用代码覆盖旧项目前的迁移保护能力优先完成。
