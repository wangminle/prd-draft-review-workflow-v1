验收：repository 层可读写思考级别数据。

### P1.4 LLM 适配层

- P1.4.1 新增 `src/app/services/thinking_adapter.py`，包含：
  - `ThinkingLevel` 枚举：`off|low|high`
  - `ThinkingAdapter` 枚举：`none|openai_reasoning|deepseek_reasoner|qwen_thinking|custom_json`
  - `build_thinking_payload(thinking_level, thinking_adapter, thinking_payload) -> dict`：统一方法，按适配类型转换成供应商参数
  - **核心原则：adapter 只在明确支持时注入字段，否则不传任何思考参数，避免未知字段导致 HTTP 400**
  - `none` 适配类型：返回空 dict（不传任何思考字段）
  - `openai_reasoning`：映射为 `reasoning_effort` 参数（low→"low", high→"high"）
  - `deepseek_reasoner`：作为可配置模板处理，默认映射仅供参考，管理员可通过 custom_json 覆盖
  - `qwen_thinking`：同上，作为可配置模板处理
  - `custom_json`：从 `thinking_payload` JSON 模板替换级别变量，管理员可自定义任意 provider 的参数映射
  - 所有内置 adapter 映射写入 `THINKING_ADAPTER_TEMPLATES` 常量字典，便于后续扩展
- P1.4.2 `stream_chat()` 签名新增可选参数 `extra_body: dict | None = None`，透传到请求体
- P1.4.3 `non_stream_chat()` 同上
- P1.4.4 `structured_chat()` 同上
- P1.4.5 `ChatApplicationService.prepare_chat_session()` 接收请求中的 `thinking_level` 运行时覆盖值；未传时使用模型默认值
- P1.4.6 `ChatApplicationService.prepare_chat_session()` 在组装 model_cfg 时调用 `build_thinking_payload()`，结果存入 `model_cfg["extra_body"]`
- P1.4.7 `chat.py` _stream 函数从请求体读取 `thinking_level`，传入 `prepare_chat_session()`，再传递 `extra_body=model_cfg.get("extra_body")` 到 `stream_chat()`
- P1.4.8 Review pipeline 启动请求读取 `thinking_level`，写入 `ReviewTask.step_details` 作为任务追溯信息，并传入 model_cfg
- P1.4.9 Review pipeline 的 `retry.py::structured_chat()` 调用也传递 `extra_body`

验收：配置为 `none` 的模型请求体不含任何思考字段；配置为 `openai_reasoning` 的模型请求体包含对应思考字段；内置映射仅供参考、custom_json 可覆盖；智能对话和需求审查都能使用请求中的 `thinking_level` 覆盖模型默认值；不会破坏现有模型调用（无未知字段注入）。

## 阶段 P2：Markdown 与 Mermaid 渲染修复

### P2.0 技术选型与边界确认

- P2.0.1 主方案确认：继续沿用 `marked + DOMPurify + mermaid`，不在本阶段切换解析栈
- P2.0.2 备选方案记录：`markdown-it + DOMPurify + mermaid` 仅作为后续插件化增强备选，本阶段不引入
- P2.0.3 fallback 边界确认：只做最小可读保底，不自研完整 Markdown 解析器
- P2.0.4 安全边界确认：所有 Markdown 主路径输出必须经 `DOMPurify` 清洗，Mermaid 错误态必须保留源码但避免注入未清洗 HTML
- P2.0.5 DOMPurify/Mermaid 边界确认：先清洗 `marked` 输出的普通 HTML 和 Mermaid 容器，再由 Mermaid 生成 SVG；禁止对 Mermaid SVG 套用仅 HTML profile 的二次清洗

验收：P2 开发前已明确“不换栈、本地优先、fallback 保底、错误可回溯”的技术边界。

### P2.1 前端依赖本地化

- P2.1.1 下载并固定版本的 `DOMPurify`、`marked`、`mermaid` 到 `src/static/vendor/`
  - `dompurify/dist/purify.min.js`
  - `marked/marked.min.js`
  - `mermaid/mermaid.min.js`
- P2.1.2 `index.html` 将 CDN `<script>` 改为本地路径 `<script src="/vendor/...">`
- P2.1.3 增加可观测的依赖加载检查：本地库加载失败时记录前端日志，避免静默降级
- P2.1.4 CDN 只作为可选 fallback（本地加载失败时动态加载 CDN）；发布验收不依赖公网 CDN
- P2.1.5 记录第三方库版本、来源 URL 和许可证信息到 `src/static/vendor/NOTICE.md`

验收：断网环境下 Markdown 主路径仍可用；阻断 CDN 域名后页面不丢失 Markdown 渲染能力。

### P2.2 Fallback Markdown 增强

- P2.2.1 聊天页 `_renderMarkdown(text)` 重构为 `_renderMarkdownWithLibraries(text)` + `_renderMarkdownFallback(text)`
- P2.2.2 Fallback 至少支持：`h1~h6`、段落、无序/有序列表、粗体、斜体、行内代码、fenced code block（含语言标记）
- P2.2.3 Fallback 识别 ` ```mermaid ` 代码块，生成 `<div class="mermaid-chart" data-mermaid="...">` 容器
- P2.2.4 fallback 不继续扩展复杂 Markdown 特性，避免用正则复刻完整解析器
- P2.2.5 统一聊天页和审查结果页的 Markdown/Mermaid 渲染策略，避免两个页面维护不同 fallback
- P2.2.6 fallback 复杂度约束：fallback 代码行数原则上不超过主渲染逻辑的 1/3；超过时优先删减 fallback 能力

验收：CDN 全部不可用时，标题、列表、代码块、Mermaid 容器仍能正确渲染。

### P2.3 Mermaid 渲染时机、异常兜底与状态样式

- P2.3.1 Mermaid 初始化统一使用 `startOnLoad: false`，禁止库自动扫描整页
- P2.3.2 流式输出过程中：只更新 Markdown 文本，不触发 Mermaid 渲染
- P2.3.3 收到 `[DONE]` 或请求结束后：对完整消息统一调用 `_renderMermaidCharts()`
- P2.3.4 SSE 异常中断兜底：在 `error`/`close`/`finally` 场景下也触发一次最终 Mermaid 渲染，避免未收到 `[DONE]` 时图表永远不出现
- P2.3.5 idle debounce 兜底：最后一块 chunk 到达后约 2 秒无新 chunk，则触发一次最终 Mermaid 渲染；同一条消息需防重复触发
- P2.3.6 渲染前先调用 Mermaid `parse` 校验，通过后再调用 `render`
- P2.3.7 DOMPurify/SVG 顺序保护：Mermaid SVG 生成前只清洗 Markdown HTML 和容器；如必须清洗 SVG，需显式允许 SVG profile，避免 SVG 被破坏
- P2.3.8 只对已闭合的 Mermaid fenced block（` ```mermaid ... ``` `）创建图表容器
- P2.3.9 `.mermaid-chart` 增加 loading 动画（渲染前显示）
- P2.3.10 Mermaid 渲染失败时保留原始源码，显示错误提示 + "查看源码" 折叠区域
- P2.3.11 `main.css` 新增对应样式规则，确保错误态在窄屏不溢出
- P2.3.12 历史消息 `_appendMessage()` 仍可立即触发 Mermaid 渲染（文本已是完整）

验收：流式过程中不出现 Mermaid 解析错误；正常结束、异常中断或 idle debounce 后 Mermaid 都能最终渲染；未闭合代码块不会提前创建图表；失败时可查看源码。

### P2.4 Mermaid 加载态与失败态样式

- P2.4.1 本阶段不作为独立交付单元；加载态、失败态、源码回显和窄屏样式并入 P2.3 同批实现
- P2.4.2 Code review 时按 P2.3 的渲染逻辑和样式一起检查，避免样式和状态 HTML 脱节

验收：P2.3 完成时同步完成 Mermaid 加载态和失败态样式，不单独拆分发布。

### P2.5 自动化测试

- P2.5.1 前端契约测试 `test_markdown_contract.py`：
  - Markdown 标题渲染（`### xxx` -> `<h3>`）
  - 列表渲染（`* item` -> `<ul><li>`）
  - Fenced code block 渲染
  - Mermaid 容器生成（` ```mermaid ` -> `[data-mermaid]`）
- P2.5.2 CDN 不可用时 fallback 渲染测试
- P2.5.3 流式过程中不提前渲染 Mermaid 的测试
- P2.5.4 SSE 异常中断或 2 秒无新 chunk 时仍触发最终 Mermaid 渲染的测试
- P2.5.5 DOMPurify 不破坏 Mermaid SVG 输出的测试或浏览器验证
- P2.5.6 Mermaid 语法错误时展示错误态并保留源码的测试
- P2.5.7 浏览器真实验证：本地启动后在 Chrome DevTools Network 面板阻断 `jsdelivr.net`，新建对话发送含标题、列表、Mermaid 代码块的消息，确认标题/列表正常渲染，Mermaid 图表在流式结束或异常兜底后出现
- P2.5.8 回归验证：全量 pytest 通过

### P3.6 审查 Pipeline 集成

- P3.6.2 **快照策略**：审查任务启动时，将公共库文档的当前 Markdown 复制一份快照到 `runtime/data/library_snapshots/{task_id}/{library_doc_id}.md`，pipeline 读取快照而非公共库原文件；这保证公共库文档后续更新不影响已生成的报告

- P3.6.5 报告输出中标注公共文档来源和 content_hash

验收：公共库文档正确参与审查；任务启动时创建快照；报告基于快照生成、不受后续更新影响；缓存 hash 包含公共文档；报告可追溯。
