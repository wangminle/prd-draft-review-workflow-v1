# 运行时数据与持久化接口规约

## 1. 文档定位

本文是项目的数据结构与持久化接口规约，用来约束后续开源版本和持续迭代：

- 先明确当前系统的数据事实：哪些数据写到哪里、以什么结构保存、哪些是事实源、哪些只是缓存或临时态。
- 再明确后端访问这些数据时应使用的 repository / storage / log writer 接口边界。
- 后续页面、服务编排、任务流、模型接入可以重构，但不得随意破坏本文定义的数据语义和兼容规则。

一句话结论：当前项目已经在目录层面完成代码与运行数据分离，分界线是工作区根目录下的 `runtime/`；后续架构治理的重点不是改变数据位置，而是把 router 中分散的数据库、文件和日志访问收口到稳定接口。

## 2. 规约原则

### 2.1 兼容性优先

1. 保持 `runtime/` 作为唯一运行时数据边界。
2. 保持现有 SQLite 表语义，不为了重构而改写事实源。
3. 新写入数据库的运行时文件路径一律保存为相对 `runtime/` 的路径。
4. 旧数据中的绝对路径、`./runtime/...`、`../runtime/...` 等历史格式由读取层兼容，不再由写入层继续制造。
5. 新增表或字段必须提供初始化迁移逻辑，保证旧数据库可升级。
6. 新增文件落盘流程必须同时定义删除、清理和缓存失效策略。

### 2.2 分层访问

后端访问数据时应逐步收敛为四层：

```text
routers
  -> application services
    -> repositories / storage / log writers / in-memory stores
      -> sqlite / runtime filesystem / jsonl logs / process memory
```

各层职责：

| 层 | 应负责 | 不应负责 |
| --- | --- | --- |
| routers | HTTP 入参、鉴权依赖、响应码、response schema | 直接 `db.add()`、`await db.commit()`、`open()`、`os.makedirs()` |
| application services | 用例编排、事务边界、状态流转、跨 repository/storage 协作 | 直接依赖 FastAPI Request/Response 细节 |
| repositories | 结构化数据查询和写入 | 文件系统访问、日志落盘、HTTPException |
| storage | runtime 文件保存、读取、删除、路径兼容 | 权限判断、业务状态流转 |
| log writers/readers | JSONL 日志写入与查询 | 反向驱动主业务状态 |
| in-memory stores | 进程内短时状态，如 SSE ticket | 伪装成长期持久化事实源 |

## 3. runtime 目录边界

`runtime/` 根路径由 `RUNTIME_ROOT` 环境变量覆盖；未覆盖时默认使用工作区根目录下的 `runtime/`。应用统一通过 `app.runtime_paths.runtime_path(...)` 计算运行时路径。

当前目录结构基线：

```text
runtime/
├── data/
│   ├── app.db
│   ├── app.db.bak-*
│   ├── converted/
│   │   └── doc_{document_id}/
│   └── review_uploads/
│       └── {project_id}/
│           ├── requirement/
│           └── historical/
├── logs/
│   ├── app.log
│   ├── audit.jsonl
│   ├── frontend.jsonl
│   ├── llm_sessions.jsonl
│   ├── server.stdout.log
│   ├── server.stderr.log
│   └── audit_backup.jsonl / audit_converted.jsonl
├── results/
├── uploads/
└── server.pid
```

目录职责：

| 路径 | 当前职责 | 主流程是否写入 |
| --- | --- | --- |
| `runtime/data/app.db` | 主业务数据库，保存结构化业务数据 | 是 |
| `runtime/data/review_uploads/` | 需求审查原始 DOCX 文件 | 是 |
| `runtime/data/converted/` | DOCX 转 Markdown 中间产物与转换缓存 | 是 |
| `runtime/uploads/` | 智能对话上传的上下文原始文件 | 是 |
| `runtime/logs/app.log` | 后端应用文本日志 | 是 |
| `runtime/logs/audit.jsonl` | 结构化审计日志 | 是 |
| `runtime/logs/frontend.jsonl` | 浏览器端上报日志 | 是 |
| `runtime/logs/llm_sessions.jsonl` | 每次 LLM 调用日志 | 是 |
| `runtime/logs/server.stdout.log` | 进程标准输出重定向 | 是 |
| `runtime/logs/server.stderr.log` | 进程标准错误重定向 | 是 |
| `runtime/results/` | 预留目录，当前未发现主流程写入 | 否 |
| `runtime/server.pid` | 启动脚本维护的进程号文件 | 是 |

## 4. session 边界

“session” 在当前系统中有三种不同含义，后续设计必须拆开。

### 4.1 认证会话

- 登录后由后端签发 JWT，保存在客户端。
- JWT 本身不落盘到 `runtime/`，数据库也没有 session 表。
- SSE 使用短时一次性 ticket，保存在进程内 `_sse_tickets` 字典，只是内存态。
- 服务重启后 JWT 仍可用，但未消费的 SSE ticket 会失效。

### 4.2 智能对话会话

- 聊天会话是持久化业务数据。
- 会话元数据保存在 `conversations`。
- 用户/助手消息保存在 `messages`。
- 会话级上下文项保存在 `chat_context_items`。
- 上下文引用的原始文件位于 `runtime/uploads/`，数据库记录 `file_id` 和抽取文本。

### 4.3 LLM 调用会话日志

- 每次 LLM 调用额外写入 `runtime/logs/llm_sessions.jsonl`。
- 这是排障和审计数据，不等价于用户可见的聊天会话。
- 后续更换模型或流程时，应保留“每次模型调用一条结构化日志”的语义。

## 5. 数据库结构规约

当前结构化数据事实源是 `runtime/data/app.db`，数据库为 SQLite。

### 5.1 用户与聊天域

| 表名 | 作用 | 关键字段 |
| --- | --- | --- |
| `users` | 用户主表 | `username` `password_hash` `role` `last_active_at` |
| `conversations` | 智能对话会话头 | `user_id` `title` `model_id` `prompt_template` |
| `messages` | 对话消息明细 | `conversation_id` `role` `content` `token_count` |
| `chat_context_items` | 会话右侧上下文项 | `conversation_id` `context_type` `file_id` `url` `manual_text` `extracted_text` `enabled` |
| `prompt_templates` | 对话 Prompt 模板 | `name` `system_prompt` `user_prompt_template` |
| `model_configs` | 模型配置 | `model_id` `api_base` `encrypted_api_key` `display_order` `deleted_by_user` |
| `skill_configs` | Skills 配置 | `skill_id` `name` `local_path` `update_url` |

### 5.2 需求审查域

| 表名 | 作用 | 关键字段 |
| --- | --- | --- |
| `review_projects` | 审查项目 | `name` `created_by` |
| `review_documents` | 项目文档主表 | `project_id` `filename` `file_path` `md_path` `content_hash` `document_type` `status` |
| `review_tasks` | 审查任务/流水线实例 | `project_id` `mode` `status` `current_step` `context_version` `model_id` `step_statuses` `step_details` |
| `doc_analyses` | 逐篇分析结果 | `document_id` `task_id` `category` `quality_score` `full_analysis` |
| `system_reviews` | 体系级审查结果 | `task_id` `project_id` `business_value` `architecture` `competition` `product_strategy` `tech_evolution` `pm_growth` `action_plan` `pm_scores` |
| `review_contexts` | 项目级评审上下文版本 | `project_id` `version` `is_active` `change_log` `context_data` |
| `review_prompts` | 评审风格 Prompt | `name` `content` `version` `is_active` |

### 5.3 全文检索

- 当前只显式维护 `messages_fts` FTS5 虚表，用于聊天消息全文检索。
- 审查文档正文和分析结果当前没有 FTS 持久化表。
- 后续如补检索层，应新增索引/投影表，不应改写现有主表语义。

## 6. 文件存储规约

### 6.1 智能对话上传文件

- 入口：上传接口。
- 落盘目录：`runtime/uploads/`
- 文件命名：`{uuid}.{ext}`
- 数据库存储：`chat_context_items.file_id` 记录文件名，必要时 `extracted_text` 记录抽取文本。
- 当前文件类型覆盖：`txt`、`pdf`、`docx`、`md`、`csv`、`json`。

这类文件是聊天会话的附属原件，不是审查项目资产。

### 6.2 需求审查上传文件

- 入口：项目文档上传接口。
- 落盘目录：`runtime/data/review_uploads/{project_id}/{document_type}/`
- 当前 `document_type` 至少包括：`requirement`、`historical`
- 文件命名：`{uuid}.docx`
- 数据库存储：`review_documents.file_path` 保存相对 `runtime/` 的路径，例如：

```text
data/review_uploads/6/requirement/2340eac7aef840c0a0d5d54a021c1d2b.docx
```

### 6.3 DOCX 转 Markdown 中间产物

- 落盘目录：`runtime/data/converted/doc_{document_id}/`
- 每个文档一个独立目录。
- 目录中至少可能包含：
  - 转换后的一个或多个 `.md`
  - `.source_hash` 源文件哈希缓存文件
  - 转换器产出的资源子目录
- 缓存命中依赖源文件 hash；源文件未变时可复用转换结果。

转换目录是可重建缓存层，不是主事实源；但当前审查流程依赖它提升效率，因此清理策略必须谨慎。

## 7. 日志规约

### 7.1 应用日志

- 文件：`runtime/logs/app.log`
- 格式：普通文本日志
- 用途：后端运行信息、告警、异常堆栈

### 7.2 审计日志

- 文件：`runtime/logs/audit.jsonl`
- 格式：每行一个 JSON
- 典型字段：

```json
{
  "timestamp": "2026-06-01T10:00:00",
  "level": "info",
  "action": "auth.login",
  "result": "success",
  "actor": {"user_id": 1, "username": "admin", "role": "admin"},
  "request": {"method": "POST", "path": "/api/auth/login", "client_ip": "127.0.0.1", "user_agent": "..."},
  "target": {"type": "frontend", "id": "review"},
  "detail": {}
}
```

审计日志是行为审计接口，不是调试输出，也不应用来反向驱动主业务状态。

### 7.3 前端日志

- 文件：`runtime/logs/frontend.jsonl`
- 来源：前端调用 `/api/log`
- 典型字段：`timestamp` `level` `page` `message` `detail`

### 7.4 LLM 调用日志

- 文件：`runtime/logs/llm_sessions.jsonl`
- 典型字段：`timestamp` `model` `input_messages` `response` `usage` `elapsed_ms` `error` `reasoning_content`（非空时写入）

### 7.5 运维输出日志

- 文件：`runtime/logs/server.stdout.log` 和 `runtime/logs/server.stderr.log`
- 来源：启动脚本或进程管理器重定向
- 作用：运维排障，不参与业务语义

`audit_backup.jsonl`、`audit_converted.jsonl` 当前应视为运维辅助产物，不作为稳定接口的一部分。

## 8. 持久化接口规约

本节定义后续重构时应收口出的接口面。方法名可以在实现时微调，但事实源边界不要改变。

### 8.1 ConversationRepository

职责：聊天会话、消息、消息搜索。

```python
class ConversationRepository(Protocol):
    async def create_conversation(self, *, user_id: int, title: str | None, model_id: str, prompt_template: str | None) -> Conversation: ...
    async def get_conversation(self, conversation_id: int, *, user_id: int) -> Conversation | None: ...
    async def list_conversations(self, *, user_id: int, page: int, page_size: int) -> PaginatedConversationRows: ...
    async def append_message(self, *, conversation_id: int, role: str, content: str, token_count: int | None = None) -> Message: ...
    async def list_messages(self, conversation_id: int, *, user_id: int) -> list[Message]: ...
    async def touch_updated_at(self, conversation_id: int) -> None: ...
    async def delete_conversation(self, conversation_id: int, *, user_id: int) -> None: ...
    async def search_messages(self, *, user_id: int, query: str, limit: int = 50) -> list[SearchHit]: ...
```

边界：

- 可封装 `messages_fts` SQL。
- 不负责拼接 LLM prompt。
- 所有权校验通过 `user_id` 参数下沉，避免散落在 router。

## 9. 应用服务建议

repository / storage 不直接表达完整业务用例，必须由 application service 组合。

| 服务 | 职责 | 主要依赖 |
| --- | --- | --- |
| `ChatApplicationService` | 新建/加载对话、保存消息、装配上下文、流结束后落库 | `ConversationRepository` `ContextItemRepository` `ChatFileStorage` `ModelConfigRepository` |

## 10. 事务与一致性规则

### 10.1 事务边界

- 单个 repository 方法默认不主动 `commit()`，由 service 控制事务边界。
- 当前历史代码依赖“写完立即可见”的场景可保留 `flush()`，但 `commit()` 应逐步收到 service。
- review pipeline 是长流程，应按步骤提交，不要跨整个 pipeline 持有巨事务。

### 10.2 文件与数据库一致性

- 文件先落盘再写库，或先写临时文件再原子移动，必须固定一种策略。
- 删除项目、删除文档这类“删库 + 删文件”场景，由 application service 统一编排。
- storage 不负责项目权限校验；权限校验在 service / repository 完成后再调用 storage。

### 10.3 路径规则

- 数据库继续保存 runtime 相对路径。
- 所有路径兼容读取逻辑只保留在 storage。
- router / service 不再自己拼接 `runtime/...` 或 `data/...`。
