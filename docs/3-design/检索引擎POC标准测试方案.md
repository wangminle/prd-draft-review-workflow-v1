# 检索引擎 POC 标准测试方案

> 本文档定义"理想的检索引擎 POC 应该怎么做"，并给出对已完成 POC-A/B 的差距分析和补测建议。
> 结论见最后一节。

---

## 1. POC 的目标边界

POC 要回答三个问题，不多也不少：

1. **哪个向量库在生产嵌入模型下质量最好？**（命中率、拒答能力）
2. **延迟和并发是否满足 5-20 人团队的需求？**（P50/P95/P99、并发 QPS）
3. **运维是否可接受？**（备份恢复、runtime 目录兼容、依赖复杂度）

POC 不需要回答：生产级高可用、大规模数据（百万 chunk）、跨地域部署、多租户隔离。这些是 P3+ 阶段的问题。

---

## 2. 标准测试方案

### 2.1 嵌入模型（最关键前提）

| 阶段 | 嵌入方案 | 问题 |
|------|---------|------|
| 已完成 POC-A/B | TF-IDF fallback（4096 维） | 词袋模型，无语义理解，向量未归一化，不代表生产行为 |
| **标准 POC 应使用** | OpenAI text-embedding-3-small（1536 维） | 生产方案，归一化密集向量，语义理解能力 |

**TF-IDF 作为 fallback 验证向量库"能跑通"是可以的，但不能作为质量评估依据。**

### 2.2 数据集

```
文档：真实业务文档（已有 runtime 45 份，943 chunks）✓
嵌入：text-embedding-3-small 真实 API 调用（POC-A/B 未做）✗
```

文档量足够：943 chunks、45 篇覆盖需求文档、规范、评审报告三类，代表性可以。无需再扩充。

### 2.3 问题集设计

标准问题集需要覆盖四种类型，难度递增：

| 类型 | 占比 | 示例 | 能否自动生成 |
|------|------|------|------------|
| 标题/章节定位 | 40% | "某需求的背景包含哪些内容" | ✓ 已有 |
| 语义匹配 | 30% | "哪个需求涉及权限控制" | 部分手写 |
| 跨文档聚合 | 20% | "哪些需求提到了数据导出" | 手写 |
| 无答案 | 10% | "系统支持离线使用吗" | 手写 |

**POC-A/B 的问题集：只有标题/章节定位 + 4 个无答案，缺少语义匹配和跨文档类型，导致命中率虚高。**

建议问题集规模：40-50 个问题（其中手写 15-20 个，自动生成 25-30 个）。

### 2.4 候选方案

只需对比以下 3 个（POC-A/B 已淘汰 Milvus Lite，sqlite-vec 作为参考）：

| 方案 | 测试原因 |
|------|---------|
| LanceDB | 当前首选，需要真实嵌入下的验证 |
| Chroma | 备选，POC-A 显示延迟比 LanceDB 低 3.5 倍（4.8ms vs 16.9ms），需要 P50/P99 确认 |
| FTS5 | 降级基准，无需重测（关键词检索行为与嵌入模型无关） |

Milvus Lite 暂不测（530ms 延迟、进程锁未验证，远期候选）。

### 2.5 评估指标

#### 检索质量

| 指标 | 计算方式 | 门槛 |
|------|---------|------|
| top-5 命中率 | 期望文档在前5结果中的比例 | ≥ 90% |
| top-1 命中率 | 期望文档排第一的比例 | ≥ 80% |
| no_answer 误答率 | 无答案问题返回"有结果"的比例 | ≤ 30% |
| 权限正确率 | 跨 workspace 零越权 | = 100% |

#### 延迟（5轮预热后测量，不含首次冷启动）

| 指标 | 门槛 |
|------|------|
| P50 | < 200ms（含 embedding API ~150ms + 检索） |
| P95 | < 500ms |
| P99 | < 1000ms |

> 注：含 embedding API 调用的端到端延迟才是真实用户体验，POC-B 测的是"纯向量库查询延迟"（不含 embedding），不等于用户等待时间。

#### 并发（ThreadPoolExecutor 测试）

| 并发数 | QPS 门槛 |
|--------|---------|
| 5 | ≥ 10 QPS（含 embedding API） |
| 10 | ≥ 15 QPS |

> 注：含 embedding API 调用后 QPS 会大幅下降（受限于 API rate limit 和网络延迟），POC-B 的 120 QPS 是纯向量库查询，不含 embedding。

#### 运维

| 指标 | 验收方式 |
|------|---------|
| 备份恢复 | 目录复制 → 重新连接 → 查询结果一致 |
| 冷启动 | 进程重启后首次查询延迟 |
| 增量写入 | 新增文档后不重建全量索引 |
| 依赖安装 | `pip install` 成功，无 native 编译问题 |

### 2.6 拒答策略验证（关键补充）

需要单独验证两种拒答方案在真实嵌入下的表现：

| 策略 | 测试方法 |
|------|---------|
| top-1/top-2 分数差 | 在 no_answer 问题上统计 `_distance[1] - _distance[0]` 的分布，找合理阈值 |
| LLM 证据判断 | 可选，在 P2 开发阶段验证，不在 POC 阶段 |

**POC-B 测试原始分数阈值（而非分数差），结论是"不可用"，但分数差策略本身没有测过。**

## 3. POC-A/B 与标准方案的差距

| 维度 | 标准方案 | POC-A/B 实际 | 差距影响 |
|------|---------|-------------|---------|
| 嵌入模型 | text-embedding-3-small | TF-IDF fallback | **高**：质量结论不可直接迁移到生产 |
| 问题类型 | 4类（含语义/跨文档） | 2类（标题+无答案） | **中**：命中率虚高，拒答能力评估不足 |
| 延迟测量 | 端到端（含 embedding API） | 仅向量库查询 | **高**：P50/P95 数据不代表用户体验 |
| Chroma 对比 | P50/P95/P99 + 并发 | 只有单轮平均延迟 | **中**：Chroma 延迟优势未确认 |
| no_answer 样本 | 10%（~5个） | 4个 | **中**：拒答策略校准依据不足 |
| 分数差拒答 | 明确测试 | 未测（测的是原始阈值） | **中**：P2.E.3 的 0.15 阈值无实验依据 |

## 4. 是否需要重跑 POC？

### 结论：**不需要完整重跑，补一个小型 POC-C 即可**

#### 不需要重跑的理由

- POC-A/B 已经回答了"哪些库能跑通、运维如何、并发能力如何（纯向量库层面）"，这些不受嵌入模型影响。
- 向量库的选择（LanceDB）主要基于运维和代码简洁性，这部分结论成立。
- 重跑完整 POC-A 浪费时间，向量库基础能力（备份、权限过滤、runtime 兼容）已验证。

#### 需要补做的 POC-C（预计 2-3 天）

**目标**：用真实嵌入模型补齐 POC-A/B 的核心缺口，不重复已有工作。

**范围**：

```
数据：runtime 45 篇真实文档，943 chunks（直接复用）
嵌入：text-embedding-3-small 真实 API（一次性费用约 $0.001）
向量库：LanceDB + Chroma（两个候选，不测其他）
问题集：复用现有 84 个 + 手写 15 个语义/跨文档问题（共 ~100 个）
```

**要回答的问题**：

1. 真实嵌入下 LanceDB top-5 命中率是多少？（预期 ≥ 95%，目前只有 TF-IDF 下的 95.2% 作为参考）
2. Chroma 在真实嵌入下延迟比 LanceDB 快多少？（4.8ms vs 16.9ms 是 TF-IDF 数据，真实嵌入可能不同）
3. 分数差拒答策略在真实嵌入下合适的阈值是多少？（no_answer 问题的 _distance 分布）
4. 端到端延迟（含 embedding API 调用）的 P50/P99 是多少？

**不在 POC-C 里做的**：Milvus Lite（已知延迟高，远期候选），sqlite-vec（已知结论），Hybrid（已知 TF-IDF 下退化，真实嵌入下留待 P2 开发中验证）。

**时间和成本**：

| 项目 | 估算 |
|------|------|
| 脚本开发（复用现有框架） | 1天 |
| 跑实验 | 半天 |
| 分析和报告 | 半天 |
| OpenAI embedding API 费用 | ~$0.001（943 chunks × 平均 200 tokens × $0.02/1M） |
| 总计 | 2天 |

---

## 5. POC-C 验收标准

POC-C 完成后，需要能回答：

- [ ] LanceDB 在真实嵌入下 top-5 命中率 ≥ 90%（否则需要重新审视选型）
- [ ] Chroma vs LanceDB 端到端延迟差异（作为最终备选决策依据）
- [ ] 分数差拒答阈值确定（更新 P2.E.3 的 0.15 默认值）
- [ ] 端到端 P99 < 1000ms（确认用户体验可接受）

如果 LanceDB top-5 < 90% 或端到端 P99 > 2000ms，需要重新讨论是否上 BGE-M3 本地嵌入（规避 API 延迟）。
