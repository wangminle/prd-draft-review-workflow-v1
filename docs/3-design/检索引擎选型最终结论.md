# 检索引擎选型最终结论

> 基于 POC-A（5 方案横向对比）和 POC-B（FTS5+LanceDB 混合检索深度验证）全部实验数据，确定 P2 检索服务的引擎选型、架构策略和拒答方案。

## 1. 实验总览

| 维度 | POC-A | POC-B |
|------|-------|-------|
| 目的 | 5 方案横向对比，筛选候选 | FTS5+LanceDB 混合检索深度验证 |
| 数据集 | 合成样本 30 份 + Runtime 真实需求 45 份 | Runtime 真实需求 45 份（同 POC-A） |
| 嵌入方式 | TF-IDF fallback（维度 4096） | TF-IDF fallback（维度 4096） |
| 评估口径 | 旧口径 + 校正口径 + Runtime 复验 | 校正口径，5 轮延迟统计 |
| 候选方案 | FTS5 / LanceDB / Milvus Lite / Chroma / sqlite-vec | FTS5 / LanceDB / Hybrid(FTS5+LanceDB) |
| 补测维度 | — | 延迟分布 P50/P95/P99、5/10/20 并发压测、无答案阈值策略、LanceDB 备份恢复 |

## 3. 关键发现

### 3.1 混合检索退化：Hybrid 92.9% < 纯 LanceDB 95.2%

POC-B 最意外的发现是 FTS5 + LanceDB 混合检索反而比纯 LanceDB 低 2.3 个百分点。经分析，退化由**两个 POC 特有问题叠加**导致，均不会出现在生产环境：

**问题一：FTS5 bigram 误召回**

FTS5 的 `unicode61` tokenizer 对中文做 bigram 切分（"需求评审" → "需求""求评""评审"），对自然语言查询产生误召回。当 FTS5 以 0.3 权重参与排序时，不相关但关键词命中的文档被推到 top-5，挤掉了 LanceDB 原本正确召回的文档。

实际影响：84 个问题中有 2 个因此退化（RQ-073、RQ-075）。

**问题二：TF-IDF fallback 向量未归一化**

POC-A/B 均使用 TF-IDF fallback 作为嵌入，TF-IDF 向量未经 L2 归一化。LanceDB 对未归一化向量使用 L2 距离（非 cosine），导致 `_distance > 1` 时 `score = 1 - _distance` 为负值。Hybrid 的 score normalization（除以 max_score）进一步放大失真，使不相关但正分的文档排在正确文档之前。

在退化案例 RQ-073 中：
- LanceDB 单独返回 doc-43（正确），`_distance=1.200`，`score=-0.200`
- Hybrid 中 doc-43 负分被排到后面，doc-46（`vec_score=3.37`，正值）排在前面

**生产环境影响评估**：

| 问题 | 生产环境是否复现 | 理由 |
|------|----------------|------|
| FTS5 bigram 误召回 | ⚠️ 可能 | 中文分词天然局限，需调权重或智能合并 |
| TF-IDF 未归一化 | ❌ 不会 | 生产使用真实嵌入模型（OpenAI/BGE-M3），向量已归一化，cosine distance ∈ [0,2] |

### 3.2 Hybrid 延迟更稳定（但代价是质量下降）

| 指标 | LanceDB | Hybrid | 改善 |
|------|---------|--------|------|
| P50 | 10.3ms | 8.5ms | -17% |
| P95 | 17.9ms | 9.8ms | -45% |
| P99 | 46.4ms | 17.2ms | -63% |

Hybrid 的 P99 大幅改善，因为 FTS5 的快速召回为长尾查询提供了低延迟备选。但质量下降 2.3% 不值得用延迟改善来换。

### 3.3 并发能力：LanceDB 120 QPS，足够 5-20 人团队

| 方案 | 并发数 | QPS | P95(ms) |
|------|--------|-----|---------|
| FTS5 | 10 | 1395 | 4.1 |
| LanceDB | 10 | 121 | 79.3 |
| Hybrid | 10 | 150 | 24.8 |

LanceDB 在 5/10/20 并发下稳定在 ~120 QPS。5-20 人团队每人每秒 1-2 次检索请求，120 QPS 完全够用。FTS5 的 1395 QPS 说明关键词检索几乎无瓶颈。

### 3.4 LanceDB 权限正确率 90.9% 是 POC 脚本问题

POC-B 中 LanceDB 的权限正确率为 90.9%（20/22），而 FTS5 和 Hybrid 均为 100%。根因：

- POC-B 的 LanceDB 检索器是一个 lambda，接受 4 个参数 `(query, workspace, top_k, role)` 但**忽略了 role 参数**
- inactive/non-member 角色查询仍返回结果，导致 2 个 role check 失败
- 20 个 workspace_id 检查全部通过，LanceDB 的 `.where(prefilter=True)` 过滤正确

**结论**：这是 POC 脚本编码疏漏，LanceDB 本身的 workspace 过滤能力为 100%。P2 生产代码将在应用层实现完整的 role + workspace 权限控制。

### 3.5 无答案拒答：POC-C 真实嵌入下 dist[0] 阈值可行 ⭐

POC-B 在 TF-IDF 下证明原始分数阈值不可用。POC-C 在真实嵌入下发现 **dist[0] 绝对阈值可行**：

可答问题与无答案问题的 dist[0] 分布存在清晰分离区间：

| 向量库 | 可答 dist[0] P90 | 无答案 dist[0] P10 | 分离间隙 |
|--------|-----------------|--------------------|---------|
| LanceDB | 1.021 | 1.033 | 0.012 |
| Chroma | 0.511 | 0.516 | 0.005 |

推荐阈值：
- **LanceDB**: dist[0] ≥ 1.0 拒答（无答案拒答率 ~80%，可答误拒率 <10%）
- **Chroma**: dist[0] ≥ 0.5 拒答

gap 分数差作为补充：
- **LanceDB**: gap < 0.065 低置信度标记
- **Chroma**: gap < 0.032 低置信度标记

### 3.6 端到端延迟瓶颈是 embedding API ⭐

POC-C 发现 LanceDB P99 = 1167ms，但这主要来自 OpenAI embedding API 的冷启动延迟（首次 HTTPS 连接 ~500-1000ms）。热连接下：
- embedding API: ~150-200ms
- LanceDB 向量查询: ~10ms
- **端到端合计: ~160-210ms**

向量库本身延迟差异（LanceDB 10ms vs Chroma 2ms）在端到端延迟中占比 <5%，不应作为选型主要依据。

|------|-----------------|----------------|
| 备份时间 | 7ms | 5ms |
| 备份大小 | 15.3MB | 6.1MB |
| 恢复后记录数 | 943 | 943 |
| 恢复后查询 | 正常 | 正常 |

## 5. 拒答策略决策

原始分数阈值方案在所有方案下均失败（误答率 25%~100%），P2 需采用替代策略。

### 5.1 推荐拒答方案（POC-C 真实嵌入数据支撑）

| 阶段 | 策略 | 原理 | 实现成本 | 预期效果 |
|------|------|------|---------|---------|
| **P2 首版** | dist[0] 绝对阈值 + gap 低置信度标记 | POC-C 证明可答/无答案的 dist[0] 有清晰分离 | 低 | 无答案拒答率 ~80% |
| **P2 增强版** | LLM 证据判断 | RAG 回答前让 LLM 判断"检索文档是否足以回答" | 中 | 减少约 90% 误答 |
| **P3 远期** | Cross-Encoder 重排序 | 交叉编码器对 top-k 重新打分 | 高 | 检索质量 + 拒答双提升 |

### 5.2 拒答策略详细设计（POC-C 校准）

**dist[0] 绝对阈值（P2 首版实现，POC-C 数据支撑）**：

```python
def should_reject(results, engine="lancedb"):
    """基于 POC-C 真实嵌入数据的 dist[0] 绝对阈值拒答。"""
    thresholds = {"lancedb": 1.0, "chroma": 0.5}
    if not results:
        return True  # 无结果 = 证据不足
    threshold = thresholds.get(engine, 1.0)
    return results[0]["_distance"] >= threshold
```

**gap 低置信度标记（P2 首版补充）**：

```python
def confidence_level(results, engine="lancedb"):
    """基于 POC-C 数据的 gap 分数差置信度标记。"""
    gap_thresholds = {"lancedb": 0.065, "chroma": 0.032}
    if not results or len(results) < 2:
        return "low"
    gap = results[1]["_distance"] - results[0]["_distance"]
    return "low" if gap < gap_thresholds.get(engine, 0.065) else "high"
```

**LLM 证据判断（P2 增强版）**：

在 RAG 回答生成前，先让 LLM 评估检索到的文档是否足以回答问题。如果 LLM 返回"证据不足"，则拒答并提示用户换一种方式提问。

这需要在现有 SkillRunner 的 prompt 中增加一个证据判断环节，实现成本中等。

### 5.3 不推荐的方案

| 方案 | 不推荐原因 |
|------|-----------|
| 原始分数阈值 | POC-B 已证明误答率 25%~100%，完全不可用 |
| FTS5 + LanceDB 交叉验证 | 两个引擎都召回才认为可信，会导致漏拒率过高（FTS5 命中率只有 42.9%） |

## 6. P2 检索服务架构建议

### 6.1 首版架构（POC-C 真实嵌入验证后更新）

```
用户查询
    │
    ▼
┌─────────────────┐
│  EmbeddingService│ ← 启动时预热连接（减少冷启动延迟）
│  embed(query)    │ ← 查询缓存（避免重复 API 调用）
└──────┬──────────┘
       │ query_vec (~150-200ms)
       ▼
┌─────────────┐    LanceDB 不可用
│  检索路由器   │ ──────────────────→ FTS5 降级检索
│  (Retriever) │
└──────┬──────┘
       │ 正常路径
       ▼
┌─────────────┐    dist[0] ≥ 1.0
│  LanceDB     │ ──────────────────→ 拒答（证据不足）
│  语义检索     │    gap < 0.065 → 低置信度标记
└──────┬──────┘
       │ top-k 结果
       ▼
┌─────────────┐
│  权限过滤     │  workspace_id + role
└──────┬──────┘
       │ 授权结果
       ▼
┌─────────────┐
│  RAG 生成    │  LLM + 检索上下文
└─────────────┘
```

### 6.2 存储路径

```
runtime/
├── data/
│   └── app.db              # SQLite 主库 + FTS5（已有）
├── vector/
│   └── lancedb/            # LanceDB 向量索引（新增）
│       ├── chunks.lance/
│       └── ...
└── config/
    └── ...
```

### 6.3 嵌入模型选择

| 选项 | 模型 | 维度 | 延迟 | 成本 | 推荐 |
|------|------|------|------|------|------|
| A | OpenAI text-embedding-3-small | 1536 | ~50ms/批 | $0.02/1M tokens | ✅ P2 首选 |
| B | BGE-M3 本地部署 | 1024 | ~100ms/批 | 免费（需 GPU/CPU） | 备选 |
| C | TF-IDF fallback | 4096 | ~1ms/批 | 免费 | 仅限离线/无网环境 |

P2 首版推荐使用 OpenAI text-embedding-3-small（**POC-C 已验证**）：
- 向量已归一化，cosine distance ∈ [0,2]，不会出现 TF-IDF 的负分问题
- 1536 维在 sqlite-vec 上限 8192 以内，未来迁移无障碍
- API 成本极低（$0.02/1M tokens），943 chunks 约 50K tokens ≈ $0.001
- **端到端延迟 ~160-210ms（热连接），需启动时预热 HTTPS 连接**
- 标题/章节定位命中率 100%（POC-C 实测）

### 6.4 索引构建时机

- 文档上传时：同步提取文本 → 切块 → 调用嵌入 API → 写入 LanceDB
- **服务启动时**：调用一次 embedding API 预热连接（减少冷启动 P99 延迟）
- 查询时：命中 `_query_cache` 时跳过 API 调用（避免重复嵌入）
- 批量重建：管理员可触发全量重建索引
- 增量更新：文档删除时标记 archived，不物理删除向量

## 7. 风险与缓解（POC-C 更新）

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|---------|
| LanceDB pyarrow 依赖冲突 | 中 | 部署受阻 | 隔离 venv；备选 Chroma/sqlite-vec |
| OpenAI API 不可用 | 低 | 无法构建新索引 | 已有索引可继续查询；TF-IDF 离线 fallback |
| embedding API 冷启动延迟高 | 中 | 首次查询 P99 > 1s | 启动预热 + 查询缓存 |
| Hybrid 质量不如预期 | 中 | 推迟混合检索上线 | 首版只用 LanceDB 单引擎，验证后再加 FTS5 |
| 拒答阈值误判 | 中 | 误拒或漏拒 | POC-C 数据支撑 dist[0] ≥ 1.0；后续迭代 LLM 证据判断 |
| 语义匹配命中率低（37.5%） | 低（首版不涉及） | 语义检索场景覆盖不足 | 首版只做标题/章节定位；后续增加 reranker |
| LanceDB 大数据量性能退化 | 低 | 检索变慢 | 当前 943 chunks 远未达瓶颈；必要时迁移 Milvus |
