# 架构分层重构 WBS 计划书

## 1. 文档定位

本文是基于 [运行时数据与持久化接口基线.md](运行时数据与持久化接口基线.md) 的实施计划书。

基线文档回答“数据结构和持久化接口如何被约束”；本文回答“在不破坏旧数据兼容的前提下，如何分阶段把当前后端改成清晰分层架构”。

目标不是一次性重写系统，而是按风险最小、收益可验证的顺序，把 router 中分散的数据库读写、文件落盘、路径兼容、JSONL 日志访问逐步收口到 repository / storage / log writer / application service。

## 2. 总体目标

### 2.1 业务目标

1. 新开源项目必须能继续基于旧项目的 `runtime/` 数据升级。
2. 后续迭代不能因为代码重构导致旧数据库、旧上传文件、旧转换缓存不可用。
3. 数据事实源保持稳定，架构层可以持续演进。

### 2.2 技术目标

1. router 只处理 HTTP 入参、鉴权依赖、响应码和 response schema。
2. application service 负责业务编排、事务边界和跨资源一致性。
3. repository 负责 SQLite 结构化数据。
4. storage 负责 runtime 文件与路径兼容。
5. log writer / reader 负责 JSONL 日志。
6. 进程内短时状态用 store 接口隔离，不伪装成数据库持久化。

## 3. 当前耦合总览

| router / 模块 | 当前混合职责 | 主要风险 |
| --- | --- | --- |
| `app/routers/upload.py` | 上传文件写磁盘、URL 抓取、文本抽取 | 文件入口散落，后续 chat storage 难复用 |
| `app/routers/chat.py` | 会话建档、消息写入、上下文项 CRUD、聊天上传文件读取、模型配置读取、流式结束后消息落库 | 流式协议和持久化混在一起，竞态修复成本高 |
| `app/routers/history.py` | 对话查询、删除、消息全文检索 | 与 chat 共用事实源但查询逻辑分散 |
| `app/routers/review.py` | 项目/文档/任务/上下文/Prompt CRUD、原始文件落盘、转换缓存管理、pipeline 持久化、缓存命中判定、文件删除、审计日志 | 路径、任务状态、LLM 流程和数据库写入高度耦合 |
| `app/routers/admin.py` | 用户、Prompt、模型、API Key、Skill 配置、审计统计读取 | 后台配置事实源分散，审计读取直接扫 JSONL |
| `app/routers/auth.py` | 登录活跃时间写回、注册写库、改密写库、审计日志 | 逻辑集中但仍直接写库 |
| `app/services/auth.py` | SSE ticket 内存字典 | 内存态未接口化 |
| `app/logging_config.py` | JSONL 日志写入 | 日志函数可用，但调用点仍散 |

## 4. 改造原则

1. 先切稳定边界，再迁移调用点。
2. 先从风险低、边界清楚的模块建立模板，再治理 `review.py`。
3. 任何阶段都不得改变 [运行时数据与持久化接口基线.md](运行时数据与持久化接口基线.md) 定义的数据事实。
4. 每个阶段必须有自动化测试和可执行验收标准。
5. 不追求一次性把所有 ORM 查询包完；只迁移当前阶段覆盖的行为。
6. 不把 SkillRunner、LLM provider、JWT 编解码混入本轮持久化分层。

## 5. WBS 总览

| WBS | 阶段 | 目标 | 主要产出 |
| --- | --- | --- | --- |
| 0 | 基线保护 | 建立迁移前安全网 | 现状测试、数据兼容检查、router 持久化扫描 |
| A | 聊天与上传模板 | 先在低风险域建立分层样板 | `ChatFileStorage`、`ConversationRepository`、`ContextItemRepository`、`ChatApplicationService` |
| B | 日志接口收口 | 把 JSONL 访问从业务调用点抽象出来 | `AuditLogWriter`、`FrontendLogWriter`、`LlmSessionLogWriter`、`AuditLogReader` |
| C | Review 文件边界 | 锁住最容易出错的路径和转换缓存规则 | `ReviewFileStorage`、路径兼容集中化 |
| D | Review pipeline 持久化 | 隔离任务状态、分析结果、体系审查落库 | `ReviewTaskRepository`、`ReviewPipelinePersistenceService` |
| E | 项目/后台/认证收口 | 完成剩余 CRUD 与配置域分层 | Project/Context/Prompt/Model/User/Skill repositories |
| F | 清理与冻结 | 删除重复路径逻辑，固化验收门槛 | router 瘦身、文档更新、回归报告 |

## 6. WBS 0：基线保护

### 0.1 建立数据兼容测试

任务：

- 0.1.1 准备最小旧库样本，覆盖用户、聊天、上下文、项目、文档、任务、分析、模型配置。
- 0.1.2 准备旧路径样本，覆盖 runtime 相对路径、`./runtime/...`、绝对路径。
- 0.1.3 验证服务启动时迁移逻辑不会破坏旧库。
- 0.1.4 验证旧上传文件和转换缓存仍能读取。

验收：

- 旧数据库可启动。
- 旧聊天记录可查询。
- 旧审查项目、文档、报告可查询。
- 旧文件路径可解析，新写入路径仍为 runtime 相对路径。

### 0.2 建立 router 持久化扫描

任务：

- 0.2.1 增加或整理脚本/测试，扫描 router 中的 `db.add()`、`await db.commit()`、`open()`、`os.makedirs()`、`shutil.rmtree()`。
- 0.2.2 初始阶段允许白名单，后续每完成一个 WBS 域就收紧该域白名单。

验收：

- 扫描结果能列出当前直接持久化访问点。
- 后续迁移不会靠人工记忆判断是否回退。

## 7. WBS A：聊天与上传分层模板

### A.1 抽 `ChatFileStorage`

范围：

- `app/routers/upload.py`
- `app/routers/chat.py` 中读取 `runtime/uploads/` 的逻辑

任务：

- A.1.1 新增 `src/app/storage/chat_file_storage.py`。
- A.1.2 封装上传文件名生成、目录创建、原始文件保存。
- A.1.3 封装文件正文读取和文本抽取协作。
- A.1.4 URL 抓取逻辑保留为独立 fetcher 或 service 协作者，不继续留在 router。
- A.1.5 更新 upload/chat 调用点。
- A.1.6 补充上传、读取、异常路径、SSRF 相关回归测试。

验收：

- `upload.py` 不再直接决定 `runtime/uploads/` 的目录和文件名。
- 聊天发送时不直接拼接上传文件路径。
- 原有上传和 URL 抓取测试通过。

### A.2 抽 `ConversationRepository`

范围：

- `app/routers/chat.py`
- `app/routers/history.py`

任务：

- A.2.1 新增 conversation repository。
- A.2.2 迁移会话创建、查询、列表、删除。
- A.2.3 迁移用户消息和助手消息写入。
- A.2.4 迁移 FTS 检索 SQL。
- A.2.5 所有权校验统一通过 `user_id` 参数下沉。
- A.2.6 补充会话列表、详情、删除、搜索、跨用户隔离测试。

验收：

- `chat.py` 和 `history.py` 不再直接操作 `Conversation` / `Message` ORM。
- `messages_fts` 的使用只出现在 repository 层。
- 聊天历史功能和搜索功能行为不变。

### A.3 抽 `ContextItemRepository`

任务：

- A.3.1 迁移上下文项列表、启用/禁用、新建、删除、清空。
- A.3.2 保留文件正文抽取在 storage/service，不放进 repository。
- A.3.3 补充上下文竞态相关前端/后端回归测试。

验收：

- `chat_context_items` 表访问只经过 repository。
- 已有上下文启用/删除/发送前同步行为不回退。

### A.4 建立 `ChatApplicationService`

任务：

- A.4.1 组合 conversation、context、file storage、model config、prompt template。
- A.4.2 router 保留 SSE/流式协议包装。
- A.4.3 service 负责新建会话、保存用户消息、装配上下文、流结束后保存助手消息。

验收：

- `chat.py` 不再承担主要持久化编排。
- SSE 行为与错误处理不变。

## 8. WBS B：日志接口收口

### B.1 抽 log writer

任务：

- B.1.1 新增 `AuditLogWriter`、`FrontendLogWriter`、`LlmSessionLogWriter` 实现。
- B.1.2 复用现有 JSONL 字段语义，不改变日志格式。
- B.1.3 将调用点从直接函数调用逐步替换为显式 writer。

验收：

- audit/frontend/llm 三类日志仍写入原 JSONL 文件。
- 新增业务入口不再直接 `open()` 写 JSONL。

### B.2 抽 `AuditLogReader`

任务：

- B.2.1 迁移 `admin.py` 中最近访问统计的 `audit.jsonl` 扫描逻辑。
- B.2.2 保留时间窗口、limit、异常行跳过行为。

验收：

- 管理后台最近访问统计结果不变。
- 日志读取和日志写入不强行塞进同一个接口。

## 10. WBS D：Review pipeline 持久化

### D.1 抽任务与结果仓储

任务：

- D.1.1 新增 `ReviewTaskRepository`。
- D.1.2 迁移任务创建、运行中任务去重、状态更新、取消、失败、完成。
- D.1.3 新增或拆出 `ReviewAnalysisRepository` 和 `SystemReviewRepository`，或在第一步先由 `ReviewTaskRepository` 承载。
- D.1.4 迁移 `doc_analyses` 写入/更新。
- D.1.5 迁移 `system_reviews` 写入/更新。
- D.1.6 迁移缓存命中查询。

验收：

- pipeline 代码不再直接拼 ORM 写 `DocAnalysis` / `SystemReview`。
- running/pending 去重、failed/cancelled/completed 状态语义不变。
- 结果页可继续读取失败任务的部分结果。

### D.2 抽 `ReviewPipelinePersistenceService`

任务：

- D.2.1 service 负责将 SkillRunner 输出转换成落库 payload。
- D.2.2 service 负责 step_statuses / step_details 的读写封装。
- D.2.3 pipeline 主流程只表达步骤执行和取消检查。

验收：

- LLM 执行逻辑和结果持久化逻辑可单独测试。
- pipeline 失败不会继续污染后续步骤状态。

## 12. WBS F：清理与冻结

### F.1 router 瘦身验收

任务：

- F.1.1 收紧 router 持久化扫描白名单。
- F.1.2 移除重复路径 helper。
- F.1.3 移除重复 JSONL 扫描逻辑。
- F.1.4 更新开发文档和测试说明。

验收：

- 已迁移 router 中不再直接出现新的 `db.add()`、`await db.commit()`、`open()`、`os.makedirs()`、`shutil.rmtree()`。
- runtime 路径转换只在 storage 层。
- 新增持久化行为必须先补接口再补实现。

### F.2 全量回归

任务：

- F.2.1 跑全量 pytest。
- F.2.2 跑关键手工链路：登录、聊天、上传上下文、项目上传、快速审查、结果查看、模型配置。
- F.2.3 用旧库样本做升级验证。

验收：

- 自动化测试通过。
- 旧数据升级通过。
- 新数据写入仍符合 [运行时数据与持久化接口基线.md](运行时数据与持久化接口基线.md)。

## 13. 第一阶段建议验收包

如果只做一个最小可交付阶段，建议范围如下：

1. WBS 0：基线保护。
2. WBS A：聊天与上传分层模板。
3. WBS B：日志接口收口。
4. WBS C.1：ReviewFileStorage。

这个范围能先解决最清晰的持久化散点和最容易继续出 bug 的路径问题，同时避免一开始就大拆 review pipeline。

## 14. 不在本计划内的事项

- 不重写 SkillRunner 编排策略。
- 不替换 LLM provider 调用层。
- 不改变 JWT 架构。
- 不改变 SQLite 为其他数据库。
- 不调整前端 UI 架构。

这些事项可以后续独立立项，但不应混入本次持久化分层重构。
