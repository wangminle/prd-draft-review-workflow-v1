## 一、整合目标

将两部分已有成果整合为统一的Web服务平台：

| 已有成果 | 形态 | 目标 |
|---------|------|------|
| AI产品需求初审（FastAPI+SQLite+SPA） | 可运行的Web应用 | 扩展后端路由和前端页面 |
| 6个审查Skills（CLI/脚本） | 可独立运行的Python脚本 | 封装为后端异步服务，通过API提供能力 |

**核心变更**：在FastAPI后端新增"审查业务逻辑层"，将Skills的CLI调用改为Python模块导入，通过REST API + SSE对外提供服务。

## 二、设计原则

1. **统一OpenAI-compatible接口**：所有LLM调用走 `app/services/retry.py` 的 `structured_chat()`，复用管理后台配置的API Key
2. **API Key复用**：审查页面直接使用管理后台配置的模型和API Key，无需单独配置
3. **异步优先**：Skills的同步代码用 `asyncio.to_thread()` 包装，不阻塞事件循环
4. **自动重试**：参考opencode的双重重试机制，对LLM调用实施指数退避+服务端Header感知
5. **上下文溢出不重试**：token超限错误走内容截断而非重试
6. **Skills保持独立**：封装层（`services/`）导入Skills的核心函数，不修改Skills源码
7. **优雅降级**：docx转换优先用Skill，Skill不可用时自动fallback到mammoth

## 五、LLM重试机制

参考opencode的两层重试架构，适配局域网内部API场景：

### 5.1 两层重试架构

```
Layer 2: 业务重试 (routers/review.py _run_pipeline)
  • Step失败后自动重试整个步骤（最多3次）
  • 上下文溢出→截断后重试

Layer 1: API重试 (services/retry.py retryable_chat)
  • 429/5xx→指数退避重试（最多5次）
  • 尊重 Retry-After Header
  • 上下文溢出→抛出ContextOverflowError（不重试）
  • 网络错误→重试
  • 4xx业务错误→不重试
```

### 5.2 退避策略

| 尝试次序 | 延迟 | 说明 |
|---------|------|------|
| 第1次重试 | 2s | `initial_delay * factor^0` |
| 第2次重试 | 4s | `initial_delay * factor^1` |
| 第3次重试 | 8s | `initial_delay * factor^2` |
| 第4次重试 | 16s | `initial_delay * factor^3` |
| 第5次重试 | 30s | 受 `max_delay_ms` 封顶 |

### 5.3 可重试 vs 不可重试

| 错误类型 | 可重试 | 说明 |
|---------|--------|------|
| HTTP 429 (Rate Limit) | 是 | 优先使用 Retry-After |
| HTTP 503 (Overloaded) | 是 | 内部API高负载 |
| HTTP 5xx (Server Error) | 是 | 服务端临时故障 |
| 网络超时/连接失败 | 是 | 瞬态网络问题 |
| 上下文溢出 | **否** | 走截断路径 |
| HTTP 4xx (业务错误) | **否** | 请求本身有问题 |

### 5.4 structured_chat

`retryable_chat()` 之上封装 `structured_chat()`，自动从LLM响应中提取JSON（支持```json块包裹、括号匹配、raw_text降级）。

## 六、API Key复用机制

审查页面**直接复用管理后台配置的模型和API Key**，无需单独配置：

```
管理后台 → 模型配置 → 填写API Base / API Key（Fernet加密存储）
                                │
                                ▼
审查工作台 → 模型选择器 → 选择已配置的模型
                                │
                                ▼
routers/review.py → _get_model_config() → 从DB读取ModelConfig → 解密API Key → 传入structured_chat()
```

**校验链**：
1. 模型是否存在（不存在→400）
2. 模型是否已启用（禁用→400）
3. API Key是否已配置（未配置→400）

### 7.3 SSE认证方案

`EventSource` API不支持自定义Header，采用query param方案：

```javascript
// api.js
getReviewProgress(projectId, reviewId) {
    const url = `/api/review/projects/${projectId}/reviews/${reviewId}`;
    const token = this.getToken();
    const params = token ? `?token=${encodeURIComponent(token)}` : '';
    return new EventSource(url + params);
}
```

```python
# review.py SSE端点
async def review_progress_sse(token: str | None = Query(None), ...):
    if not token:
        return StreamingResponse(...)  # unauthorized
    try:
        decode_token(token)
    except JWTError:
        return StreamingResponse(...)  # unauthorized
```

## 九、流水线调度逻辑

`routers/review.py` 的 `_run_pipeline()` 实现了异步5步调度：

1. **Step 0 预处理**：调用 `docx-to-markdown` Skill 的 `convert_docx_to_markdown()`（`asyncio.to_thread`包装），Skill不可用时自动fallback到mammoth
2. **Step 1 分类**：调用 `structured_chat()` 让LLM识别文档分类和版本号
3. **Step 2 逐篇分析**：逐篇调用 `structured_chat()` 执行6维度分析，结果存入 `doc_analyses` 表
4. **Step 3+ 体系Review/PM评估/报告生成**：根据mode决定执行哪些步骤
5. **SSE进度推送**：每步状态变更写入 `asyncio.Queue`，SSE端点消费推送

**容错**：单篇文档分析失败不影响其他文档，最终状态为 `completed`（部分成功）或 `failed`。
