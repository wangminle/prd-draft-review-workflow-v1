# 快速审核专家意见维度功能说明

## 1. 功能背景

在单篇快速审核场景中，原有 `per_analysis` 更偏向需求文本的基础理解，核心输出聚焦在：

- 核心问题
- 分类
- 边界定义
- 边界外问题
- 要点提取
- 质量评分

这套输出能够回答“文档写了什么”，但对“文档是否符合团队长/专家在需求评审中的关注口径”支持不够直接。前期讨论中已经明确，快速审核需要额外补上一组更贴近审查经验的判断维度，用来回答：

- 需求范围是不是写实
- 边界是不是讲完整
- 分类和权益是不是结构化
- 用户侧命名是不是可理解
- 多入口文案是不是一致
- 技术分期是不是清楚

因此，FEAT-026 的目标不是重做快速审核流程，而是在现有单篇分析结果中新增一个独立的“专家意见维度”结果块，让快速审核除了给出基础分析之外，还能直接给出更接近专家评审口径的结论和修改建议。

## 2. 功能目标

本功能的目标有 4 个：

1. 在不改 SkillRunner 编排的前提下，为快速审核补充一组单独展示的专家意见结果。
2. 让 `per_analysis` 一次输出中同时包含基础分析和专家意见分析，减少新增步骤带来的复杂度。
3. 让后端 API、前端结果页和 Markdown 后备报告都能透传并展示这组结果。
4. 把专家经验沉淀为结构化规则，便于后续继续扩展、调权重或演进为独立审查步骤。

## 3. 设计原则

### 3.1 先改 Skill，不改 Runner

本次实现明确采用“先改 Skill 输出结构，再沿既有链路透传”的方案，不新增新的 pipeline step。原因有三点：

- 快速审核当前本来就通过 `per_analysis` 完成单篇核心分析，新增字段可以直接复用这一步的上下文。
- SkillRunner 当前对 `per_analysis` 的结果是透明存储，新增字段不会破坏既有调度模式。
- 如果把专家意见拆成单独步骤，会同时增加缓存、取消、重试、报告聚合和前端状态管理的复杂度，当前收益不匹配。

也就是说，本功能属于“丰富单篇分析结果”，而不是“新增一条编排链路”。

### 3.2 专家意见必须单列

专家意见维度不能混在核心问题、边界或要点提取里，否则前端和使用者无法直观看到“这是一组独立的审查结论”。因此本次使用独立字段：

```json
"expert_review": {
  "summary": "整体结论",
  "checks": []
}
```

## 4. 专家意见维度定义

`expert_review` 固定覆盖以下 6 条规则：

| rule_key | 规则名称 | 审查关注点 |
|---|---|---|
| `scope_realism` | 需求范围要写实 | 是否明确说明当前需求到底解决什么，而不是只写价值背景 |
| `boundary_completeness` | 能力边界要写全 | 是否说明做什么、不做什么、依赖条件和前置限制 |
| `structured_entitlements` | 权益和分类要结构化 | 是否把用户权益、对象分类、场景分类讲清楚 |
| `user_facing_naming` | 用户侧命名要可理解 | 是否使用用户能理解的名称，而不是内部术语 |
| `copy_consistency` | 多入口文案要统一 | 是否存在不同入口、页面、账号体系下命名不一致 |
| `phased_tech_plan` | 技术方案要分期但不能糊涂 | 是否写清阶段边界、适用范围和当前落点 |

每条规则固定输出以下字段：

- `rule_key`：规则唯一标识
- `rule_name`：规则名称
- `status`：检查结果，只允许 `pass` / `risk` / `missing`
- `evidence`：来自文档的依据；如果文档没写，要明确说明“文档未体现”
- `suggestion`：具体补充或改写建议

同时，整个专家意见块还会输出一个 `summary`，用 1 到 2 句话总结该文档在专家意见维度上的整体成熟度。

## 5. 端到端实现链路

### 5.1 Skill Prompt 与输出 Schema

`prd-per-analysis` 的 prompt 现在新增第 7 个分析维度，明确要求模型：

- 单独输出 `expert_review`
- 严格覆盖 6 条规则
- 每条规则都必须给出依据和建议
- 不允许把专家结论混入其它维度

与此同时，输出 schema 将 `expert_review` 升级为必填字段，确保模型输出结构稳定、可被后续链路消费。

### 5.2 后端 API 透传

后端在 `AnalysisInfo` 中新增 `expert_review` 字段，并在 `/analyses` 接口中从 `full_analysis` 里提取该块数据后返回。这样快速审核和结果聚合接口无需新增独立查询接口，就能直接把专家意见透传到前端。

这意味着：

- 数据库存储仍然复用 `full_analysis`
- API 层只做字段提取，不额外改动持久化模型结构
- 历史分析结果如果没有 `expert_review`，前端会自然降级为不展示该区块

### 5.3 前端结果展示

前端在逐篇分析卡片中新增“专家意见维度”区块，位置放在边界定义之后、规范合规之前。展示内容包含：

- 整体结论
- 六项规则逐条检查结果
- 每项规则的状态徽标
- 依据与建议

状态展示约定如下：

- `pass`：通过
- `risk`：有风险
- `missing`：缺失

这样用户在看快速审核结果时，可以直接把基础分析和专家评审结论并排理解，而不需要从长段文字中自行归纳。

### 5.4 Markdown 后备报告

如果最终报告没有命中已润色的 `report_markdown`，后备 Markdown 报告也会补充专家意见内容，至少输出：

- 专家意见结论
- 六项规则逐条状态
- 每条规则的依据与建议

这保证了导出路径和页面展示路径在信息上尽量一致，不会出现“页面有，导出没有”的断层。

```json

  "expert_review": {
    "summary": "范围基本清晰，但文案统一和阶段边界仍需补齐。",
    "checks": [
      {
        "rule_key": "scope_realism",
        "rule_name": "需求范围要写实",
        "status": "pass",
        "evidence": "已明确当前需求聚焦统一预约入口。",
        "suggestion": "保持当前写法。"
      },
      {
        "rule_key": "copy_consistency",
        "rule_name": "多入口文案要统一",

        "suggestion": "补充 App、活动页和后台入口的统一命名表。"
      }
    ]
  }

```

## 7. 本次落地范围

本次 FEAT-026 已覆盖以下内容：

- `per_analysis` Prompt 新增专家意见维度定义
- 输出 schema 新增 `expert_review` 必填约束
- 单篇分析脚本模型结构补齐 `expert_review`
- 后端 `AnalysisInfo` 新增 `expert_review`
- `/analyses` 接口透传 `expert_review`
- 逐篇分析前端单独展示“专家意见维度”
- Markdown 后备报告补充专家意见结论与逐条检查
- SkillRunner、前端契约、报告契约测试补齐

## 8. 验收与测试

本功能在落地时已完成两层验证：

### 8.1 定向验证

围绕本次改动新增或扩展了以下测试：

- SkillRunner 集成测试：验证 `per_analysis` 输出里的 `expert_review` 能进入 pipeline state
- 前端契约测试：验证逐篇分析区存在独立的专家意见渲染逻辑
- 报告契约测试：验证 `/analyses`、`/report` 和 Markdown 后备报告都能带出 `expert_review`

定向 pytest：69 通过。

### 8.2 全量回归

全量 pytest：201 通过。

这说明本次改动没有破坏快速审核已有的基础分析、报告输出和结果页渲染链路。

## 9. 当前边界与后续演进

当前实现仍有几个明确边界：

1. 专家意见仍然属于 `per_analysis` 的一部分，尚未成为独立可编排步骤。
2. 当前没有单独的专家意见评分汇总卡，也没有进入概览页统计。
3. 历史老数据如果没有 `expert_review`，结果页不会回填，只会按旧格式展示。
4. 专家意见结果目前主要服务于单篇快速审核，尚未单独进入体系 Review 的聚合逻辑。

后续如果出现以下需求，再考虑升级为独立步骤会更合理：

- 需要对专家意见单独缓存、重跑、重试
- 需要在概览页做跨文档统计
- 需要把专家意见维度接入 full/review 模式的聚合结论
- 需要按规则维度输出整体风险排名或看板

## 10. 总结

FEAT-026 的本质，是把“专家经验”从口头审查标准，沉淀为快速审核中的结构化输出。它没有改变快速审核的主流程，但显著提升了结果的可读性、可执行性和可沉淀性。

对用户来说，快速审核现在不只是告诉他“文档写了什么”，还会更直接地告诉他“文档还缺什么、为什么缺、下一步该怎么改”。这也是该功能的核心价值。
