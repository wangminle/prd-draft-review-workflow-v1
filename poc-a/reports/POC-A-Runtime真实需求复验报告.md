# POC-A Runtime 真实需求检索复验报告

> 本报告基于本机 runtime 保留的历史需求 Markdown 进行复验，用于校正上一版 POC-A 手写样本报告中的评估偏差。  
> 原始复验结果位于 `runtime/poc-a-real/`，该目录为运行时派生数据，按数据与代码分离原则不纳入 git。

## 1. 复验结论

这次复验不支持“Milvus Lite 因召回差而淘汰”的结论。  

在真实需求 Markdown 上，LanceDB、Milvus Lite、Chroma 的检索质量完全一致：

- top-5 命中率均为 95.2%
- top-1 命中率均为 89.3%
- 权限过滤正确率均为 100.0%

Milvus Lite 当前主要问题不是召回质量，而是本机 Lite 模式下平均查询延迟更高：530.0ms。这个延迟仍在可接受范围内，但明显高于 LanceDB 的 16.9ms 和 Chroma 的 4.8ms。因此，Milvus Lite 应保留为 Phase 2 候选，不应按上一版报告直接淘汰。

## 2. 数据来源与样本规模

复验脚本：`poc-a/scripts/runtime_eval.py`

数据来源：

- `runtime/data/app.db` 中的 `review_documents` 元数据
- 本机旧 runtime 转换目录中 `review_documents.md_path` 指向的 Markdown 正文

过滤规则：

- 排除 `status = failed` 的文档
- 排除 `md_path` 缺失或文件不存在的文档
- 排除正文长度过短的占位文件
- 按 `content_hash` 去重，避免重复上传文档影响评估

最终复验规模：

| 有效文档数 | 45 |

## 3. 测试集生成方式

测试集由脚本从真实文档自动生成，避免手写样本中的 `expect_ids` 与实际 `source_id` 不一致问题。

生成规则：

1. 每篇文档生成标题定位问题，例如“某需求主要需求是什么”。
2. 每篇文档选取有效章节生成章节定位问题，例如“某需求的需求背景包含哪些内容”。
3. 追加 4 个无答案问题，用于观察拒答/阈值能力。
4. 每个问题使用真实 `source_id` 作为期望命中文档。
5. 每个问题绑定原文档所属 `workspace_id`，用于检查权限隔离。

这版测试集修正了上一版报告的核心缺陷：不再使用人工写入的 `NORM-xxx` 期望 ID，而是直接使用索引中的真实文档 ID。

## 5. 指标解读

### 5.1 FTS5

FTS5 速度极快，平均 0.4ms，权限隔离也稳定。但在真实需求文档中，标题定位和章节定位命中率只有 40% 左右。原因是当前问题更偏自然语言表达，且真实需求文档标题和章节中存在版本号、业务词、同义表达、表格内容和长句描述，关键词检索容易漏召回。

FTS5 仍适合作为混合检索中的关键词快速召回层，但不适合单独承担语义检索。

### 5.2 LanceDB

LanceDB 在本次真实文档复验中表现稳定，质量与 Chroma/Milvus Lite 一致，平均延迟 16.9ms，明显优于 Milvus Lite。它的优势是本地目录式存储、依赖相对可控、与 `runtime/` 数据隔离模型匹配。

如果 Phase 2 的目标是尽快落地本地单机向量检索，LanceDB 仍是稳妥候选。

### 5.3 Milvus Lite

Milvus Lite 的召回质量与 LanceDB、Chroma 完全一致，说明上一版报告里“延迟过高因此淘汰”的结论过早。

本次延迟 530.0ms，高于 LanceDB 和 Chroma，但已经比上一版 1582ms 低很多。上一版延迟偏高的主要原因之一是查询时重复重建 chunk、TF-IDF 和 vocab；本次已改为预计算 encoder 后再查询。

Milvus Lite 仍需补测：

- 热查询和冷查询分开统计
- 多次重复查询的 P50/P95/P99
- 并发查询下的本地服务稳定性
- 文件锁和备份恢复边界
- 后续迁移到 Milvus Standalone/Distributed 的成本

### 5.4 Chroma

Chroma 本次平均延迟最低，质量也与 LanceDB/Milvus Lite 一致。但 Chroma 依赖链更重，历史 POC 中数据目录也更大。它适合作为生态成熟的备选方案，但是否作为默认底座，要看项目对依赖体积、长期维护和部署可控性的要求。

## 6. 无答案问题说明

三套向量库在无答案问题上的命中率为 0.0%，这不是向量库本身的检索失败，而是当前评估脚本还没有加入相似度阈值、rerank 阈值或拒答策略。

向量库默认会返回最相近结果，即使问题本身不在知识库中。因此，后续 RAG 方案必须单独设计：

- 最低相似度阈值
- top-k 分数差阈值
- FTS 与向量结果交叉验证
- LLM 回答前的“证据不足”判断
- 无答案专项测试集

无答案能力不能只靠向量库选择解决。

## 7. 推荐决策（最终版，POC-B 完成后更新）

> ⚠️ 本节已根据 POC-B 深度验证结果更新。最终选型结论详见 `docs/3-design/检索引擎选型最终结论.md`。

POC-B 验证发现 FTS5 + LanceDB 混合检索反而比纯 LanceDB 低 2.3 个百分点（92.9% < 95.2%），且原始分数阈值拒答方案完全失败。

因此，P2 检索服务的最终选型为：

```
首选：LanceDB 单引擎（95.2% top-5 / 89.3% top-1 / P50=10ms）
降级：FTS5（LanceDB 不可用时回退）
拒答：top-1/top-2 分数差 + LLM 证据判断
嵌入：OpenAI text-embedding-3-small（替代 TF-IDF fallback）
```

不建议 P2 首版直接引入 Hybrid（当前证据不支持 Hybrid 优于纯 LanceDB）。

向量库候选保留为：
1. **LanceDB** ✅ 首选
2. **Chroma** 🔄 备选（需简化依赖栈时评估）
3. **sqlite-vec** 🔄 轻量备选（单文件存储，极快）
4. **Milvus Lite** 🔄 远期候选（服务化部署时评估）
